# MoE (Mixture-of-Experts) — 적게 켜서 크게 쓰는 구조

> 학습용 문서. 요약 → 상세 → "알면 AI를 더 잘 쓰는 법" 순. 인용은 검증된 arXiv만.

## 요약 (3줄)

- 거대한 모델 전체를 매번 다 쓰면 비용이 폭발한다.
- MoE는 **여러 '전문가(expert)' 중 토큰마다 일부만 켠다** → 전체 파라미터는 크지만
  **토큰당 연산은 거의 일정**.
- 그래서 최신 고성능 오픈모델 상당수가 MoE (DeepSeek, Mixtral 등).

## 상세

### 1. 핵심 아이디어 — 희소 활성화(sparse activation)
보통(dense) 모델은 모든 파라미터가 매 토큰에 관여한다. MoE는 FFN 블록을
여러 **전문가**로 나누고, **라우터(router)** 가 토큰마다 적합한 소수(예: 64개 중 2개)만 켠다.
```
토큰 → 라우터 → [전문가 중 top-k 선택] → 그 전문가만 연산
→ 총 파라미터 ↑↑ , 토큰당 연산 ≈ 그대로
```

### 2. 대표 연구
- **Switch Transformer** [Fedus 외 2021, arXiv:2101.03961](https://arxiv.org/abs/2101.03961)
  — top-1 라우팅으로도 품질 유지, dense 대비 **사전학습 ~7배 빠름**.
- **Mixtral** [Jiang 외 2024, arXiv:2401.04088](https://arxiv.org/abs/2401.04088)
  — 토큰마다 전문가 **2개** 선택하는 오픈 MoE.
- **DeepSeekMoE** [2024, arXiv:2401.06066](https://arxiv.org/abs/2401.06066)
  — **공유 전문가 분리** + **세분화된 전문가**로 전문성↑. DeepSeek-V3로 production 규모 확장.

### 3. 장점과 함정
- 장점: 같은 추론 비용으로 **더 큰 지식 용량**. 학습·추론 효율↑.
- 함정:
  - **로드 밸런싱** — 특정 전문가에만 토큰이 쏠리면 나머지가 논다 (보조 손실로 분산 유도).
  - **메모리** — 안 켜도 전문가 파라미터는 **다 적재**해야 함 → VRAM 요구 큼.
  - **라우팅 불안정** — 같은 입력이 다른 전문가로 가면 결과가 흔들릴 수 있음.

## 알면 AI를 더 잘 쓰는 법 (어필 포인트)

> **핵심: 구조를 알면 "왜 이 모델이 빠른데 똑똑한지 / 어디서 한계가 오는지"를 안다.**

- **모델 스펙 읽는 눈**: "총 671B / 활성 37B" 같은 표기를 보면 **MoE구나, 추론은
  37B급 비용이구나**를 즉시 해석. (dense 70B와 비용·성격이 다름)
- **셀프호스팅 판단**: MoE는 활성 파라미터가 작아도 **전체를 VRAM에 올려야** 하므로,
  로컬 구동 시 메모리 산정을 활성이 아닌 **총 파라미터 기준**으로 한다.
- **일관성 기대치 조정**: 라우팅 특성상 같은 질문에도 미세하게 답이 흔들릴 수 있음 →
  중요한 작업은 temperature를 낮추고 **재현성**을 따로 확인.
- **비용 직관**: "왜 거대 모델인데 API가 의외로 싸지?"의 답이 MoE(희소 활성화)인 경우가 많다.
