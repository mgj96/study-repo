# 딥다이브 — 메모리 계층과 캐시 (자바 렌즈)

> 관점: "코드는 CPU가 실행한다"는 착각을 버린다. **진짜 병목은 데이터를 CPU까지 나르는 일**이다.
> Big-O가 같아도 `ArrayList`가 `LinkedList`를 이기고, 필드 배치 하나로 멀티스레드 성능이 10배 갈린다. 그 정체를 파본다.

## 0. 30초 직관 — 책상·서랍·창고·도서관

당신이 코드를 짜는 개발자라 치자. 지금 손에 든 펜은 **레지스터**다. 즉시 쓴다(0ns).

- **책상 위**에 펼쳐둔 자주 보는 자료 = **L1 캐시**. 고개만 돌리면 있다(1ns).
- **책상 서랍** = **L2**. 서랍 여는 시간(4ns).
- **옆 사람과 공유하는 캐비닛** = **L3**. 일어나서 다녀온다(15ns).
- **복도 끝 창고** = **메인 메모리(RAM)**. 걸어갔다 온다(100ns). L1의 **약 100배**.
- **다른 건물 지하 서고** = **SSD**. 엘리베이터 타고 간다(100µs = 100,000ns).
- **시내 도서관에 신청해서 배송받기** = **HDD 디스크 시크**. 며칠 걸린다(10ms = 10,000,000ns). RAM의 **약 10만 배**.
- **해외에 편지로 문의** = **네트워크 왕복(대륙 간)**. 몇 주(150ms).

핵심 감각 하나만 가져가라: **한 계단 내려갈 때마다 대략 10~100배씩 느려진다.** CPU가 아무리 빨라도, 창고까지 걸어갔다 오는 100ns 동안 **명령어 수백 개를 놀리며 기다린다.** 그래서 잘 짠 코드는 "연산을 줄인 코드"가 아니라 **"책상 위(캐시)에서 일이 끝나게 만든 코드"** 다.

> 자바 렌즈: `new`로 객체를 흩뿌리는 건, 자료를 창고 여기저기 아무 선반에나 던져놓는 것과 같다. 나중에 하나 찾을 때마다 창고를 헤맨다(포인터 추격). 배열에 값을 촘촘히 담는 건, **한 선반에 몰아 담아 한 번에 책상으로 가져오는 것**과 같다.

## 1. 계층 전체 지도 — 숫자로 느끼기

레이턴시는 하드웨어마다 다르지만, **비율(자릿수)** 은 수십 년째 안정적이다. 아래는 업계에서 통용되는 대표값(Jeff Dean 계열 "Latency Numbers", Colin Scott 인터랙티브 표 기준).

| 계층 | 대표 레이턴시 | 초로 환산(1ns=1초라면) | 용량 급 | 관리 주체 |
|---|---|---|---|---|
| 레지스터 | ~0.3 ns | 0.3초 | ~수백 바이트 | 컴파일러 |
| **L1 캐시** | ~1 ns | 1초 | ~32–64 KB | 하드웨어 |
| L2 캐시 | ~4 ns | 4초 | ~256KB–1MB | 하드웨어 |
| L3 캐시(공유) | ~15 ns | 15초 | ~8–32 MB | 하드웨어 |
| **메인 메모리(RAM)** | ~100 ns | **약 1분 40초** | ~수십 GB | OS/하드웨어 |
| SSD 랜덤 읽기 | ~100 µs | 약 1.2일 | ~수백 GB–TB | OS |
| **HDD 디스크 시크** | ~10 ms | **약 4개월** | ~TB | OS |
| 같은 리전 네트워크 왕복 | ~0.5 ms | 약 6일 | — | — |
| 대륙 간 네트워크 왕복 | ~150 ms | 약 5년 | — | — |

**꼭 외울 두 비율:**
- **RAM은 L1보다 약 100배 느리다** (~100ns vs ~1ns).
- **디스크 시크는 RAM보다 약 100,000배 느리다** (~10ms vs ~100ns).

> "1ns를 1초로 늘려보기"는 Drepper와 여러 강의가 쓰는 고전적 직관 장치다. L1에서 자료 꺼내는 게 1초라면, HDD에서 꺼내는 건 **4개월**이다. 왜 디스크 I/O 한 번을 그렇게 무서워하는지 몸으로 와닿는다.

```text
   빠름/작음                                                    느림/큼
   [레지스터] < [L1] < [L2] < [L3] < [RAM] < [SSD] < [HDD] < [네트워크]
      0.3ns   1ns   4ns   15ns  100ns  100µs  10ms   ...
        └──── 하드웨어가 알아서 ────┘ └──────── OS/개발자가 관리 ────────┘
```

계층이 존재하는 이유는 경제학이다. 빠른 메모리(SRAM)는 비싸고 작고 뜨겁다. 느린 메모리(DRAM, 낸드, 자기디스크)는 싸고 크다. **작고 빠른 걸 앞에, 크고 느린 걸 뒤에** 두고, "자주 쓰는 건 앞으로 당겨온다"는 도박(캐싱)으로 평균을 빠르게 만든다. 이 도박이 통하는 근거가 바로 **지역성**이다(→ 3장).

## 2. 메모리 월(Memory Wall) — CPU는 대부분 기다린다

1980년대부터 **CPU 속도는 연 ~50%씩** 빨라졌지만 **DRAM 레이턴시는 연 ~7%** 밖에 못 따라왔다. 이 격차가 벌어져 벽이 된 것이 **메모리 월(memory wall)** 이다(Wulf & McKee, 1995).

결과: 오늘날 CPU 한 클럭이 ~0.3ns인데 RAM 접근은 ~100ns. 즉 **캐시 미스 한 번 = 명령어 수백 개를 실행할 수 있는 시간을 통째로 날림.** 현대 프로세서가 실제 워크로드에서 시간의 상당 부분을 "메모리 대기(stall)"로 보낸다는 건 잘 알려진 사실이다.

그래서 하드웨어는 온갖 꼼수를 쓴다:
- **다단계 캐시**(L1/L2/L3)로 히트율을 올리고,
- **프리페처(prefetcher)**: "다음에 이 근처를 읽겠지" 예측해 미리 당겨온다(→ 순차 접근이 빠른 또 하나의 이유),
- **비순차 실행(OoO)**: 미스로 멈춘 동안 뒤 명령어를 먼저 처리,
- **하이퍼스레딩**: 한 코어가 멈춘 시간에 다른 스레드를 태운다.

> 자바 렌즈: JIT가 아무리 코드를 최적화해도 **데이터 배치가 나쁘면 프리페처와 캐시가 헛돈다.** "느린 자바"의 상당수는 JIT 문제가 아니라 **객체가 힙에 흩어져 생기는 캐시 미스**다.

## 3. 지역성(Locality) — 캐싱이 통하는 이유

캐시라는 도박이 이기는 건 실제 프로그램이 두 습관을 갖기 때문이다.

- **시간 지역성(temporal locality)**: 방금 쓴 데이터를 곧 또 쓴다. → 루프 카운터 `i`, 자주 부르는 필드. "방금 책상에 올린 자료는 잠시 안 치운다."
- **공간 지역성(spatial locality)**: 방금 쓴 데이터 **바로 옆**을 곧 쓴다. → 배열 순차 순회. "이 자료 옆 페이지도 곧 볼 테니 같이 가져온다."

캐시는 이 둘을 하드웨어로 베팅한다. 특히 공간 지역성 때문에 캐시는 **바이트 하나가 아니라 덩어리째** 가져온다. 그 덩어리가 **캐시 라인**이다.

---

## 4. 캐시 라인(~64바이트) — 모든 것의 단위

CPU는 메모리를 1바이트씩 읽지 않는다. **캐시 라인(cache line, 보통 64바이트)** 단위로 통째로 읽는다. `int` 하나(4B)를 건드려도, **그 int가 속한 64바이트 블록 전체**가 캐시로 올라온다.

이 사실 하나가 성능의 절반을 설명한다.

```text
메모리:  [ 64B 라인 A ][ 64B 라인 B ][ 64B 라인 C ] ...
int[] arr를 arr[0]부터 순회하면:
  arr[0] 읽는 순간 → 라인 A(= arr[0..15], int 16개)가 통째로 캐시로!
  arr[1..15]는 이미 캐시에 있음 → 15번은 공짜(캐시 히트)
  16개마다 한 번만 RAM에 감(미스)
```

- **순차 배열 순회**: 미스 1번당 값 16개(64B/4B)를 공짜로 처리. 게다가 프리페처가 다음 라인까지 미리 당겨온다. → **거의 캐시 히트**.
- **포인터 추격(pointer chasing)**: 매 노드가 힙 여기저기에 있어 **매번 새 라인 미스**. 프리페처도 "다음 주소"를 예측 못 한다(주소가 데이터에 들어있어 읽기 전엔 모름).

> 자바 렌즈: `int[]`는 값 16개가 한 라인에 촘촘. 하지만 `Integer[]`는 **참조(포인터) 배열** — 배열 자체는 촘촘해도, 실제 `Integer` 객체들은 힙에 흩어져 있어 값을 꺼내려면 매번 점프한다. 이게 **오토박싱이 성능에 나쁜 진짜 이유**(할당 비용 + 캐시 미스).

## 5. 왜 배열 순회가 링크드 리스트 추격을 이기나

Big-O로는 둘 다 순회 O(n)인데, 실측은 배열이 **수 배~수십 배** 빠르다. 이유는 전적으로 캐시다.

```text
배열 int[]:      [a0 a1 a2 a3 a4 a5 ...]   ← 메모리에 연속, 한 라인에 여러 개
                  ─── 순차, 프리페치 잘 됨, 미스 적음 ───▶

링크드 리스트:   node1 ─▶(힙 저 멀리)─▶ node3 ─▶(또 저 멀리) ...
                 매 노드 = 별도 객체 = 별도 라인 = 매번 미스 + 다음 주소 예측 불가
```

각 리스트 노드는 자바에서 **별도 힙 객체**다: 값 + `next` 참조 + **객체 헤더(12~16B)**. 즉 `long` 하나(8B) 담자고 노드 하나가 **32~40B**를 먹고, 그마저 힙 곳곳에 흩어진다. 64B 라인을 가져와도 **쓸모 있는 값은 8B뿐**, 나머지는 헤더·포인터 낭비.

### ArrayList vs LinkedList — Big-O가 같아도

|  | ArrayList | LinkedList |
|---|---|---|
| 임의 인덱스 접근 | O(1) | O(n) |
| 순차 순회 | O(n), **캐시 친화** | O(n), **매 노드 미스** |
| 중간 삽입/삭제 | O(n) (이동) | O(1)*(노드를 이미 쥐고 있을 때만) |
| 메모리 밀도 | 값만 촘촘 | 노드마다 헤더+포인터 오버헤드 |

교과서는 "중간 삽입 많으면 LinkedList"라 하지만, 현실에선 **거의 항상 ArrayList가 이긴다.** 이유:
1. 순회·검색이 캐시 친화적이라 압도적으로 빠르고,
2. LinkedList의 O(1) 삽입은 "삽입 지점 노드를 이미 쥐고 있을 때"뿐 — 보통은 **먼저 O(n)으로 찾아가야** 하고, 그 탐색이 캐시 미스 지옥이다.

> 실전 결론: 자바에서 `LinkedList`를 쓸 이유는 거의 없다. `ArrayDeque`가 큐/스택도 더 빠르다(→ [data-structures-deep.md](data-structures-deep.md)).

## 6. 실습 ① — 행 우선 vs 열 우선 2D 순회

자바의 2D 배열 `int[N][N]`은 **행(row) 배열들의 배열**이다. 즉 한 행 내부는 메모리에 연속이지만, 행과 행 사이는 별개 객체다. 그래서 **행 우선(row-major)** 순회는 캐시 라인을 알뜰히 쓰고, **열 우선(column-major)** 은 매 접근이 다른 행 = 다른 라인으로 튀어 미스를 쏟아낸다.

```java
public class RowVsColumn {
    static final int N = 8192;

    public static void main(String[] args) {
        int[][] a = new int[N][N];

        // 행 우선: a[i][0], a[i][1], ... 연속 주소 → 캐시 라인 알뜰히 사용
        long t1 = System.nanoTime();
        long sum1 = 0;
        for (int i = 0; i < N; i++)
            for (int j = 0; j < N; j++)
                sum1 += a[i][j];
        long rowMs = (System.nanoTime() - t1) / 1_000_000;

        // 열 우선: a[0][j], a[1][j], ... 매번 다른 행(다른 라인)으로 점프 → 미스 폭발
        long t2 = System.nanoTime();
        long sum2 = 0;
        for (int j = 0; j < N; j++)
            for (int i = 0; i < N; i++)
                sum2 += a[i][j];
        long colMs = (System.nanoTime() - t2) / 1_000_000;

        System.out.println("행 우선(row-major): " + rowMs + " ms");
        System.out.println("열 우선(col-major): " + colMs + " ms  (보통 수 배 느림)");
        System.out.println(sum1 + sum2); // JIT가 루프를 지우지 못하게 결과 사용
    }
}
```

**같은 연산 횟수, 같은 O(N²)** 인데 열 우선이 흔히 **수 배** 느리다. 유일한 차이는 **메모리 접근 순서 = 캐시 히트율**이다. (C처럼 진짜 연속 2D 블록이면 격차가 더 벌어진다. 자바는 배열-of-배열이라 조금 완화되지만 방향성은 동일하다.)

> 교훈: 알고리즘을 안 바꾸고 **루프 순서만 바꿔도** 크게 빨라질 수 있다. "연산 수"가 아니라 **"접근 순서"** 를 본다.

## 7. 캐시 연관성(associativity)과 축출(eviction)

라인을 캐시 어디에 놓을지 규칙이 있다.

- **직접 사상(direct-mapped)**: 라인마다 놓일 칸이 딱 하나. 단순하지만, **서로 다른 주소가 같은 칸을 다투면** 계속 쫓아내며 미스(충돌 미스).
- **완전 연관(fully-associative)**: 아무 칸에나. 유연하지만 검색 비용 큼(비현실적).
- **N-way 집합 연관(set-associative)**: 절충안. 라인이 갈 수 있는 칸이 N개(보통 4·8·16-way). 실제 CPU가 쓰는 방식.

한 집합이 꽉 차면 하나를 **축출(eviction)** 한다. 정책은 대체로 **LRU 근사**(가장 오래 안 쓴 것). 자바 개발자에겐 익숙한 그림이다 — **`LinkedHashMap`의 access-order LRU 캐시**가 바로 이 아이디어를 소프트웨어로 옮긴 것.

주의할 패턴: **정확히 2의 거듭제곱 스트라이드**로 큰 배열을 건너뛰며 접근하면, 접근들이 **같은 집합에 몰려** 캐시 용량이 남아도 서로를 축출한다(충돌 미스). 6장에서 `N=8192`처럼 큰 2의 거듭제곱을 쓴 것도 이 효과를 도드라지게 하려는 것.

## 8. 실습 ② — 거짓 공유(False Sharing)

멀티스레드에서 가장 음흉한 캐시 함정. **두 스레드가 서로 다른 변수를 쓰는데도** 그 변수들이 **같은 64B 캐시 라인**에 얹혀 있으면, 하드웨어 캐시 일관성(coherence) 프로토콜이 라인을 통째로 관리하기 때문에 **한 스레드의 쓰기가 다른 코어의 라인을 무효화**한다. 논리적으로 안 겹치는데 물리적으로 겹쳐서 서로를 방해한다 — 그래서 "거짓" 공유.

```text
        캐시 라인 하나(64B)
  [  long x  ][  long y  ][ ...padding... ]
     ▲코어0가 x만 씀     ▲코어1이 y만 씀
  코어0이 x 쓰면 → 라인 전체 dirty → 코어1의 캐시 라인 무효화 → 코어1은 RAM 다녀와야
  (반대도 마찬가지) → 두 코어가 라인을 핑퐁 → 병렬인데 직렬보다 느려짐
```

```java
public class FalseSharing {
    static final int THREADS = 4;
    static final long ITERS = 200_000_000L;

    // 나쁜 버전: 카운터들이 한 배열에 연속 → 같은/인접 캐시 라인 공유
    static class BadCounters { volatile long[] c = new long[THREADS]; }

    // 좋은 버전: 각 카운터를 별도 셀로 만들고 패딩으로 한 라인 독차지
    // (@jdk.internal.vm.annotation.Contended 는 JVM 옵션 -XX:-RestrictContended 필요)
    static final class PaddedCell {
        volatile long value;
        long p1, p2, p3, p4, p5, p6, p7; // 56B 패딩 → value가 64B 라인 독점
    }

    public static void main(String[] args) throws Exception {
        // --- 나쁜 버전 ---
        BadCounters bad = new BadCounters();
        run("false-sharing (같은 라인)", i -> {
            for (long n = 0; n < ITERS; n++) bad.c[i]++;
        });

        // --- 좋은 버전 ---
        PaddedCell[] good = new PaddedCell[THREADS];
        for (int i = 0; i < THREADS; i++) good[i] = new PaddedCell();
        run("padded (라인 분리)", i -> {
            for (long n = 0; n < ITERS; n++) good[i].value++;
        });
    }

    interface Job { void run(int idx); }

    static void run(String label, Job job) throws Exception {
        Thread[] ts = new Thread[THREADS];
        long t = System.nanoTime();
        for (int i = 0; i < THREADS; i++) {
            final int idx = i;
            ts[i] = new Thread(() -> job.run(idx));
            ts[i].start();
        }
        for (Thread th : ts) th.join();
        System.out.println(label + ": " + (System.nanoTime() - t) / 1_000_000 + " ms");
    }
}
```

패딩 버전이 흔히 **수 배** 빠르다. 스레드들은 서로의 데이터를 건드리지 않는데도, **캐시 라인을 공유했다는 사실 하나만으로** 병렬성이 증발했던 것.

### 해법: 패딩과 @Contended

- **수동 패딩**: 위처럼 더미 필드로 채워 핫 필드가 라인을 독점하게 한다. 단, JIT/컴파일러가 미사용 필드를 재배치·제거할 수 있어 취약하다.
- **`@jdk.internal.vm.annotation.Contended`** (JDK 8+ 내부 애노테이션): JVM이 **자동으로 앞뒤에 패딩**을 붙여 필드를 독립 라인에 배치. 애플리케이션 코드에서 쓰려면 `-XX:-RestrictContended` 필요(내부 API라 조심).
- **실전 사례**: JDK 자신이 이걸 쓴다. `LongAdder`/`Striped64`의 셀, `ForkJoinPool`, `ConcurrentHashMap`의 카운터 셀 등이 `@Contended`로 거짓 공유를 피한다. **`AtomicLong` 여러 개를 여러 스레드가 각각 증가시키는 경합 상황에선 `LongAdder`가 훨씬 빠른데**, 그 비결의 큰 축이 바로 셀 분산 + 거짓 공유 회피다(→ [gc-gotchas.md](gc-gotchas.md)의 동시성 함정과 함께 보라).

> 자바 렌즈 요약: "동기화도 안 겹치고 `volatile`도 각자인데 왜 안 빨라지지?"의 90%는 거짓 공유다. 프로파일러의 **메모리 스톨/캐시 미스** 지표를 봐야 보인다.

## 9. 데이터 지향 설계(DOD) — AoS vs SoA

객체지향의 기본형은 **AoS(Array of Structs)** — "객체들의 배열". 자바 관용구 그대로다.

```java
// AoS: 객체(참조) 배열. 각 Particle은 힙 어딘가의 별도 객체.
class Particle { float x, y, z, vx, vy, vz; boolean alive; String name; }
Particle[] world = new Particle[1_000_000];
```

물리 업데이트에서 `x`만 순회하며 건드려도, 캐시 라인엔 `y,z,vx,...,name 참조`까지 **안 쓰는 필드가 딸려 온다.** 게다가 각 `Particle`은 별도 객체라 힙에 흩어져 **순회가 곧 포인터 추격**. 라인 이용률이 처참하다.

**SoA(Struct of Arrays)** — "필드별 원시 배열"로 뒤집는다.

```java
// SoA: 필드마다 원시 타입 배열. x만 쓰는 루프는 x[] 라인을 100% 활용.
class ParticleSystem {
    float[] x = new float[N], y = new float[N], z = new float[N];
    float[] vx = new float[N], vy = new float[N], vz = new float[N];
    boolean[] alive = new boolean[N];
}
// x += vx * dt 같은 루프가 x[], vx[]를 순차로 훑음 → 캐시 히트 + 프리페치 + SIMD 친화
for (int i = 0; i < N; i++) x[i] += vx[i] * dt;
```

이것이 **데이터 지향 설계(data-oriented design)** 의 핵심: **"어떤 데이터를 함께 순회하는가"에 맞춰 메모리 레이아웃을 짠다.** 게임 엔진·수치 계산·컬럼형 DB가 다 이 길을 간다.

| | AoS (객체 배열) | SoA (원시 배열 묶음) |
|---|---|---|
| 캐시 이용률(일부 필드만 순회) | 낮음(안 쓰는 필드 딸려옴) | 높음(쓰는 배열만 촘촘) |
| 포인터 추격 | 많음(객체 참조 배열) | 없음(원시 값 연속) |
| 한 엔티티 전체 접근 | 편함 | 여러 배열을 인덱스로 조합 |
| 가독성/객체지향성 | 좋음 | 떨어짐 |
| SIMD/벡터화 | 어려움 | 쉬움 |

> 자바의 한계와 미래: 자바는 오랫동안 **참조 배열밖에** 없어 진짜 연속 struct 배열을 못 만들었다(`Integer[]`가 흩어지는 이유). **Project Valhalla의 값 클래스(value class)** 가 이걸 바꾸려 한다 — 값 타입 배열이 헤더·참조 없이 **평평하게(flat)** 깔리면 C의 struct 배열처럼 캐시 친화적이 된다. DOD를 자바에서 언어 차원으로 당겨오는 시도.

## 10. 같은 원리가 DB 인덱스에서 다시 나온다 — 페이지 = 디스크 블록

메모리 계층 사고는 DB에서 **한 계단 아래**로 똑같이 반복된다. CPU↔RAM에서 "캐시 라인(64B)"이 전송 단위였듯, RAM↔디스크에서는 **페이지(page, 보통 4KB·8KB·16KB)** 가 전송 단위다. 페이지 = 디스크 블록 하나.

- DB는 데이터를 **페이지 단위로** 디스크에서 읽어 **버퍼 풀(RAM)** 에 캐싱한다. (버퍼 풀 = DB판 캐시.) 디스크 시크가 RAM보다 10만 배 느리니, **"몇 페이지를 읽느냐"가 곧 성능**이다.
- 그래서 인덱스는 이진 탐색 트리가 아니라 **B-트리/B+트리**를 쓴다. 노드 하나를 **한 페이지 크기**로 맞춰, 한 번 디스크 읽기로 **수백 개 키**를 가져온다(넓고 얕은 트리 → 디스크 접근 횟수 최소화). 이것이 캐시 라인 = "미스 1번에 값 16개"의 디스크판이다.
- **B+트리는 리프를 연결 리스트로** 이어 **범위 스캔**이 페이지 순차 읽기가 되게 한다 → 공간 지역성 활용. `WHERE age BETWEEN 20 AND 30`이 빠른 이유.
- **커버링 인덱스**가 빠른 것도 같은 논리: 필요한 컬럼이 인덱스 페이지 안에 다 있으면 **테이블 페이지를 추가로 안 읽어도 된다**(랜덤 I/O 절약).

```text
CPU ↔ RAM :  전송 단위 = 캐시 라인(~64B),   "미스" 1번에 값 16개
RAM ↔ 디스크: 전송 단위 = 페이지(~8KB),      "페이지 폴트" 1번에 레코드 수백 개
              └── 같은 계층 사고, 스케일만 다름 ──┘
```

> 자바 렌즈: JVM의 **`ByteBuffer`/`MappedByteBuffer`(mmap)**, OS의 **페이지 캐시**, DB **버퍼 풀** 모두 "느린 계층 앞에 빠른 캐시를 두고 블록 단위로 나른다"는 **동일한 패턴**의 변주다. 계층 하나를 이해하면 나머지가 다 같은 얼굴로 보인다.

## 11. 실전 체크리스트 — 코드에 바로 적용

- **원시 배열을 우선**하라: `int[]` `long[]` `double[]` `>` `Integer[]`/객체 배열. 오토박싱은 할당 + 캐시 미스 이중고.
- **함께 순회할 데이터는 함께 배치**하라(연속 배열). 큰 객체 그래프를 매 프레임 포인터 추격하지 마라.
- **루프 순서 = 메모리 순서**로 맞춰라(행 우선). 알고리즘 안 바꾸고 공짜로 빨라진다.
- **핫한 공유 데이터**는 거짓 공유를 의심하라. 경합 카운터엔 `LongAdder`, 필요하면 `@Contended`/패딩.
- **자료구조 선택은 Big-O만 보지 말고 캐시 행동을 보라**: 보통 `ArrayList`/`ArrayDeque` `>` `LinkedList`, 밀집 `HashMap` `>` 흩뿌린 트리(순회가 잦다면).
- **DB는 "읽는 페이지 수"로 사고**하라: 인덱스로 랜덤 I/O를 순차/최소로, 커버링 인덱스로 테이블 접근 제거.
- **측정하라**: 캐시 효과는 직관을 배신한다. JMH로 벤치, 프로파일러의 캐시 미스/메모리 스톨 지표를 확인.

## 용어 사전

| 용어 | 한 줄 정의 |
|---|---|
| 캐시 라인(cache line) | CPU↔메모리 전송 최소 단위. 보통 64바이트. 한 바이트를 읽어도 라인 전체가 올라온다. |
| 시간 지역성 | 방금 쓴 데이터를 곧 또 쓰는 경향. |
| 공간 지역성 | 방금 쓴 데이터의 인접 주소를 곧 쓰는 경향. |
| 캐시 히트/미스 | 원하는 데이터가 캐시에 있음/없음. 미스는 아래 계층까지 다녀와야 한다. |
| 프리페처(prefetcher) | 다음에 읽을 주소를 예측해 미리 캐시로 당겨오는 하드웨어. 순차 접근에 강함. |
| 메모리 월(memory wall) | CPU 속도와 메모리 레이턴시의 격차가 벌어져 메모리가 병목이 되는 현상. |
| 연관성(associativity) | 한 라인이 캐시의 어느 칸에 놓일 수 있는지의 자유도(N-way). |
| 축출(eviction) | 캐시가 꽉 찼을 때 라인 하나를 내보내는 것. 대개 LRU 근사. |
| 포인터 추격(pointer chasing) | 참조를 따라 흩어진 객체를 한 개씩 방문. 매번 캐시 미스 유발. |
| 거짓 공유(false sharing) | 다른 변수인데 같은 캐시 라인에 얹혀 코어끼리 라인을 핑퐁하며 느려짐. |
| @Contended | 필드 앞뒤에 패딩을 넣어 독립 캐시 라인에 두는 JVM 내부 애노테이션. |
| AoS / SoA | 객체 배열(Array of Structs) vs 필드별 배열(Struct of Arrays) 레이아웃. |
| 데이터 지향 설계(DOD) | 접근 패턴에 맞춰 메모리 레이아웃을 설계하는 방법론. |
| 페이지(page) | RAM↔디스크 전송 단위(≈4~16KB). = 디스크 블록. DB 인덱스 노드 크기의 기준. |
| 버퍼 풀(buffer pool) | DB가 디스크 페이지를 캐싱하는 RAM 영역. "DB판 캐시". |
| B+트리 | 노드를 페이지 크기로 맞춘 넓고 얕은 인덱스 트리. 디스크 접근 횟수 최소화 + 범위 스캔 친화. |
