## 0. 30초 직관 — 책 뒤 색인, 전화번호부

책에서 "트랜잭션" 이야기를 찾고 싶다. 방법은 두 가지다.

- **1쪽부터 끝까지 넘기며 눈으로 스캔** → 500쪽이면 500쪽 다 봐야 할 수도. 이게 **풀 테이블 스캔 O(n)**.
- **책 뒤 색인(index)에서 "ㅌ" 찾기 → "트랜잭션 … 287쪽"** → 바로 287쪽으로 점프. 이게 **인덱스 조회 O(log n)**.

핵심은 색인이 **정렬돼 있어서** "ㅌ" 근처로 몇 번 만에 좁혀진다는 것. 그리고 색인은 **공짜가 아니다** — 책 뒤에 추가 페이지를 붙여야 하고(저장 공간↑), 본문을 고칠 때마다 색인도 같이 고쳐야 한다(쓰기 느려짐). DB 인덱스가 딱 이 색인이다. **전화번호부**도 같은 원리: 이름순으로 정렬돼 있으니 "김"으로 시작하는 사람을 순식간에 찾지만, 반대로 "010-1234로 시작하는 번호의 주인"은 못 찾는다(정렬 기준이 아니니까 → 뒤에 나올 **left-prefix 규칙**의 씨앗).

> 자바 렌즈: `TreeMap`(정렬 유지, 범위검색 O(log n))을 **디스크에 얹은 버전**이 B+트리 인덱스다. 다만 "노드 하나 = 디스크 페이지 하나"라는 제약이 모든 설계를 바꾼다(§3).

## 1. 인덱스란 무엇인가 — 원본을 안 건드리는 보조 구조

**인덱스는 테이블 원본과 별개로 존재하는, 정렬된 보조 자료구조**다. `(검색키, 원본 행을 가리키는 포인터)` 쌍을 검색키 순으로 정렬해 둔 것.

```text
테이블 USERS (원본, 저장 순서는 뒤죽박죽)          인덱스 idx_age (age 기준 정렬 + 포인터)
┌────┬────────┬─────┐                            ┌─────┬──────────┐
│ id │ name   │ age │  ← rowid로 접근             │ age │ → rowid  │
├────┼────────┼─────┤                            ├─────┼──────────┤
│ 17 │ Kim    │ 41  │                            │ 22  │ → row#88 │
│ 88 │ Lee    │ 22  │                            │ 30  │ → row#42 │
│ 42 │ Park   │ 30  │                            │ 41  │ → row#17 │
└────┴────────┴─────┘                            └─────┴──────────┘
```

**효과와 비용의 교환**:

- **효과**: `WHERE age = 30` 을 처리할 때, 정렬 안 된 원본은 전 행을 다 훑어야 하지만(**풀 스캔 O(n)**), 정렬된 인덱스는 이진탐색 비슷하게 좁혀 **O(log n)** 에 도달한다.
- **비용 1 — 저장 공간**: 인덱스는 별도 구조라 디스크를 더 쓴다. 컬럼마다 인덱스를 만들면 테이블보다 인덱스가 더 커지기도 한다.
- **비용 2 — 쓰기 느려짐**: `INSERT`/`UPDATE`/`DELETE` 때마다 **원본 + 관련 인덱스 전부**를 갱신해야 한다(§8 쓰기 증폭).

> 한 줄 요약: **인덱스 = 읽기를 O(n)→O(log n)으로 사는 대신, 저장 공간과 쓰기 성능을 지불하는 거래.** 그래서 "일단 다 걸자"가 아니라 "무엇을 얼마나 걸까"가 설계 문제가 된다.

## 2. 왜 하필 B+트리인가 — 이진탐색트리도, 메모리 균형트리도 아닌 이유

"O(log n)이면 이진탐색트리(BST)나 레드-블랙 트리도 되잖아?" 맞다. 순수 계산량만 보면 그렇다. 문제는 **데이터가 메모리가 아니라 디스크에 있다**는 것이다.

### 2-1. 병목은 연산이 아니라 디스크 접근이다 (메모리 계층)

컴퓨터의 저장장치는 계층이다. 위로 갈수록 빠르고 작다.

```text
CPU 레지스터   ~1 ns
L1/L2 캐시     ~1–10 ns
RAM(메인메모리) ~100 ns
SSD           ~100,000 ns  (10만 배 느림)
HDD 디스크    ~10,000,000 ns (천만 배 느림, 회전·탐색)
```

DB 인덱스가 상대하는 데이터는 대개 RAM에 다 안 들어가서 **디스크에 있다**. 그러면 알고리즘의 성능을 좌우하는 건 "비교를 몇 번 하느냐"가 아니라 **"디스크에서 페이지를 몇 번 읽어 오느냐(random I/O 횟수)"** 다. 디스크는 바이트 단위가 아니라 **페이지(보통 4KB~16KB) 단위**로 읽는다.

### 2-2. 이진 트리의 치명적 약점 — 나무가 너무 높다

이진탐색트리/레드-블랙 트리는 **노드 하나에 키 하나, 자식 둘**이다. 그래서 100만 개면 높이가 `log₂(1,000,000) ≈ 20`. 각 노드가 디스크의 서로 다른 위치에 흩어져 있다면, **한 번 탐색에 디스크 random read를 최대 20번** 해야 한다. 20 × 10만 ns = 심각하게 느리다.

### 2-3. B+트리의 해법 — 넓고 낮게 (high fanout)

B+트리는 **노드 하나에 키를 수백 개** 담는다. 노드 하나 = 디스크 페이지 하나에 꽉 채워 넣는 것이다. 그러면 자식 수(=**fanout, 분기수**)가 수백이 되어, 나무가 **믿기 힘들 만큼 낮아진다**.

- fanout이 500이라면: 1레벨=500개, 2레벨=25만, 3레벨=1.25억, **4레벨=625억 개**를 커버.
- 즉 **수백만~수억 행이라도 트리 높이는 3~4레벨**. 곧 **디스크 페이지 읽기 3~4번**이면 어떤 행이든 도달한다.
- 게다가 위쪽 1~2레벨(루트·상위 내부노드)은 자주 쓰여 **RAM 캐시(버퍼 풀)에 상주**한다. 그러면 실제 디스크 접근은 사실상 **1~2번**으로 줄어든다.

```text
이진트리:  높이 20, 노드당 키 1개  →  디스크 읽기 ~20회
B+트리:    높이 3~4, 노드당 키 수백개 →  디스크 읽기 3~4회 (캐시로 사실상 1~2회)
```

> 그래서 "메모리에서는 레드-블랙 트리, 디스크에서는 B+트리"다. **B+트리의 존재 이유 전체가 "디스크 페이지 읽기 횟수를 최소화"** 라는 한 문장으로 압축된다(→ 이 발상은 [data-structures-deep.md](data-structures-deep.md) §3의 "균형 트리는 왜 안 쏠리나"를 디스크 세계로 확장한 것).

## 3. B+트리의 구조 — 데이터는 전부 리프에, 리프끼리는 연결

B+트리는 세 종류의 노드로 이뤄진다.

- **루트 노드**: 트리의 꼭대기. 보통 RAM에 항상 떠 있다.
- **내부 노드(internal)**: **길잡이 키(separator)와 자식 포인터만** 있다. 실제 데이터는 없다. "이 값 이상은 오른쪽으로" 하는 이정표 역할.
- **리프 노드(leaf)**: **모든 키와 데이터(또는 행 포인터)가 여기에만** 있다. 그리고 결정적으로 **리프끼리 링크드 리스트로 연결**돼 있다.

### 텍스트 도식 — B+트리 (age 인덱스, fanout 예시)

```text
                       [ 30 | 60 ]              ← 루트(내부 노드): 길잡이 키만
                      /     |      \
             ┌───────┘      │       └────────┐
        [10 | 20]        [40 | 50]        [70 | 80]     ← 내부 노드: 데이터 없음
        /   |   \        /   |   \        /   |   \
       ▼    ▼    ▼      ▼    ▼    ▼      ▼    ▼    ▼
   [5 8]→[12 18]→[22 27]→[33 38]→[44 55]→[62 68]→[72 90]  ← 리프: 실제 키+행포인터
     └─────┴──────┴──────┴──────┴──────┴──────┴─────→ (다음 리프로 링크, 정렬 순서 유지)
```

두 가지가 이 구조의 핵심이다.

1. **모든 실제 데이터가 리프에만 있다** → 어떤 키를 찾든 **항상 루트에서 리프까지 같은 깊이**를 내려간다. 탐색 비용이 균일하다(모든 조회가 3~4번 페이지 읽기).
2. **리프가 정렬 순서대로 링크로 연결돼 있다** → **범위 검색이 폭발적으로 빨라진다**. `age BETWEEN 22 AND 55` 는 "22가 있는 리프까지 트리를 한 번 타고 내려간 뒤(O(log n)), **거기서부터 링크를 따라 옆으로 쭉 읽으면** 끝"이다. 다시 위로 올라갔다 내려오는 일이 없다.

이 링크드 리프 덕분에 **`BETWEEN`, `>`/`<`, `ORDER BY`, `LIMIT`(정렬 후 상위 N개)** 가 전부 인덱스만으로 순차 읽기가 된다.

> 자바 렌즈: 리프의 링크드 리스트 = `LinkedHashMap`이 삽입/정렬 순서를 유지하려고 노드를 이중 연결 리스트로 엮는 것과 같은 발상. "정렬 순회를 O(1)씩 옆으로" 하려는 목적이 동일하다.

## 4. B-트리 vs B+트리 — 데이터를 어디에 두느냐

이름이 비슷해서 자주 헷갈린다. 차이는 **"데이터를 내부 노드에도 두느냐"** 하나로 갈린다.

| | **B-트리** | **B+트리** (DB 인덱스 표준) |
|---|---|---|
| 데이터 위치 | 내부 노드에도 데이터 저장 | **리프에만** 데이터 저장 |
| 내부 노드 | 키 + 데이터 + 포인터 | **키(길잡이) + 포인터만** → 더 많은 키를 한 페이지에 → **fanout↑, 트리 더 낮음** |
| 리프 연결 | 없음 | **링크드 리스트로 연결** → 범위검색 강력 |
| 단일 키 조회 | 운 좋으면 내부에서 조기 종료 | 항상 리프까지 내려감(대신 균일) |
| 범위/정렬 스캔 | 트리를 오르내려야 해 비효율 | 리프 링크로 순차 읽기 → 최적 |

거의 모든 관계형 DB(MySQL InnoDB, PostgreSQL, Oracle)의 인덱스가 **B+트리**를 쓰는 이유가 이 표에 다 있다: **내부 노드에서 데이터를 빼내 fanout을 극대화**하고(더 낮은 트리 = 더 적은 디스크 읽기), **리프를 연결해 범위 검색을 순차 I/O로** 만들기 때문.

## 5. 클러스터형 vs 보조(비클러스터형) 인덱스 — 테이블 자체가 인덱스일 수도

인덱스 리프에 "무엇을" 넣느냐에 따라 종류가 갈린다.

### 클러스터형 인덱스 (Clustered / 대개 Primary Key)

- **테이블의 실제 행 데이터 자체가 인덱스 리프에 정렬돼 저장**된다. 즉 인덱스 = 테이블. 별도 원본이 따로 없다.
- 그래서 테이블당 **딱 하나만** 가능하다(물리적 정렬 순서는 하나뿐이니까).
- MySQL **InnoDB는 Primary Key가 곧 클러스터형 인덱스**다. PK로 찾으면 리프에 도달하는 순간 행 데이터도 거기 있다 → **추가 조회 없음**.

### 보조(비클러스터형) 인덱스 (Secondary / Non-clustered)

- 리프에 데이터가 아니라 **"어느 행인지 가리키는 값"** 만 있다. InnoDB에서는 그게 **PK 값**이다.
- 그래서 보조 인덱스로 찾으면 두 단계가 된다:
  1. 보조 인덱스 B+트리를 타고 내려가 **PK 값**을 얻고,
  2. 그 PK로 **클러스터형 인덱스를 다시 타서** 실제 행을 가져온다.
- 이 2단계를 **"테이블 되짚기(back to table)"** 또는 **북마크 룩업**이라 부른다. 조회가 사실상 두 배 일이 되는 지점이다. 이걸 없애는 게 §7 커버링 인덱스.

```text
보조 인덱스 조회 (예: WHERE name = 'Lee')
  idx_name B+트리 ── 'Lee' → PK=88 얻음
                              │
                              ▼  (되짚기: PK로 다시 탐색)
  클러스터형(PK) B+트리 ── 88 → 실제 행 {88, Lee, 22} 획득
```

> 자바 렌즈: 클러스터형 = 값 자체를 담은 컬렉션, 보조 인덱스 = "키 → 인덱스(PK)"만 담고 실제 객체는 원본에서 다시 꺼내는 구조. 되짚기 한 번이 곧 랜덤 I/O 한 번이라, "보조 인덱스를 탔는데도 느리다"의 흔한 원인이 된다.

## 6. 복합(다중 컬럼) 인덱스와 left-prefix 규칙 — 전화번호부의 정렬

여러 컬럼을 묶어 하나의 인덱스로 만들 수 있다: `INDEX idx (last_name, first_name, age)`. 이때 정렬은 **"먼저 last_name으로, 같으면 first_name으로, 또 같으면 age로"** 라는 **사전식(lexicographic) 순서**다. 전화번호부가 성 → 이름 순으로 정렬된 것과 똑같다.

여기서 **left-prefix(맨 왼쪽 접두어) 규칙**이 나온다: **선두 컬럼부터 연속으로 주어질 때만 인덱스를 탈 수 있다.**

`(last_name, first_name, age)` 인덱스 기준:

| 쿼리 조건 | 인덱스 사용? | 이유 |
|---|---|---|
| `WHERE last_name = 'Kim'` | O (선두 사용) | 전화번호부에서 "김"씨 찾기 — 정렬 선두 |
| `WHERE last_name='Kim' AND first_name='Minsu'` | O (앞 2개) | "김민수"까지 좁힘 |
| `WHERE last_name='Kim' AND age=30` | 일부만 | last_name으로 좁힌 뒤 age는 그 안에서 필터(부분 활용) |
| `WHERE first_name='Minsu'` | **X (선두 건너뜀)** | 성을 모른 채 "민수"만? 전화번호부 전체를 다 봐야 함 → 풀 스캔 |
| `WHERE age = 30` | **X** | 세 번째 컬럼만 단독 → 정렬 무의미 |

> 핵심 직관: 복합 인덱스는 **"성으로 먼저 정렬된 하나의 전화번호부"** 다. 성을 안 주면(=선두 컬럼을 안 주면) 그 정렬을 써먹을 수가 없다. 그래서 **가장 자주 단독으로 조회되는 컬럼, 또는 등치(=) 조건에 쓰이는 컬럼을 선두에** 두는 게 복합 인덱스 설계의 제1원칙이다(→ *Use The Index, Luke*의 핵심 주제).

주의: **범위 조건(`>`, `<`, `BETWEEN`)이 걸린 컬럼 이후의 컬럼은 인덱스로 좁히지 못한다.** 예를 들어 `WHERE last_name='Kim' AND first_name > 'M' AND age=30` 에서, `first_name`에 범위가 걸리는 순간 그 뒤 `age`는 인덱스 정렬을 못 쓴다(리프를 옆으로 훑으며 필터만). 그래서 **등치 조건 컬럼을 앞, 범위 조건 컬럼을 뒤**에 두는 게 유리하다.

## 7. 커버링 인덱스 — 테이블을 아예 안 읽는다

`SELECT` 가 필요로 하는 **모든 컬럼이 인덱스 안에 이미 들어 있으면**, DB는 리프에서 값을 다 얻고 **원본 테이블을 되짚지 않는다**. 이걸 **커버링 인덱스(covering index)**, 그런 조회를 **index-only scan**이라 한다. §5의 "되짚기 랜덤 I/O"를 통째로 없애는 최적화다.

```sql
-- 인덱스: INDEX idx_cover (last_name, first_name)
SELECT first_name           -- 필요한 컬럼이 전부 인덱스 안에 있음
FROM   users
WHERE  last_name = 'Kim';    -- → 리프만 읽고 끝. 테이블 접근 0회 (커버링)

SELECT first_name, email     -- email은 인덱스에 없음
FROM   users
WHERE  last_name = 'Kim';    -- → email 가지러 매 행마다 테이블 되짚기 (커버링 아님)
```

> 자바 렌즈: JPA에서 엔티티 전체(`SELECT e FROM User e`) 대신 **필요한 필드만 뽑는 DTO 프로젝션**(`SELECT new Dto(e.firstName) ...`)을 쓰면, 그 필드들이 인덱스에 있을 때 커버링 스캔으로 떨어져 훨씬 빠르다. "전체 엔티티 로딩"이 습관이 되면 이 최적화를 스스로 막는 셈.

## 8. 선택도(selectivity)와 인덱스를 "안" 쓰는 경우 — 옵티마이저의 판단

인덱스를 걸어도 옵티마이저가 **일부러 안 쓰고 풀 스캔을 고를 때**가 있다. 비합리가 아니라 **더 빠르기 때문**이다.

**선택도(selectivity)** = 그 컬럼 값이 얼마나 고유한가. 매칭되는 행이 적을수록 선택도가 높다(좋다).

- **고선택도** 예: `email`, `user_id` — 한 값에 보통 1행. 인덱스가 최고로 효과적.
- **저선택도** 예: `gender`('M'/'F'), `is_deleted`(true/false) — 한 값에 전체의 50%가 걸림.

`WHERE gender = 'M'` 이 전체의 50%를 반환한다고 하자. 인덱스를 쓰면 **매칭된 50만 행마다 되짚기 랜덤 I/O**를 해야 한다. 그럴 바엔 테이블을 **처음부터 끝까지 순차(sequential) 읽기**하는 게 낫다 — 순차 I/O는 랜덤 I/O보다 수십 배 빠르니까. 그래서 옵티마이저는 **"매칭 행이 대략 전체의 20~30%를 넘으면 인덱스를 버리고 풀 스캔"** 하는 판단을 한다(임계치는 DB·통계마다 다름).

**인덱스가 무시되는 흔한 함정들** (자바 개발자가 자주 밟는):

```sql
-- (1) 컬럼에 함수/연산을 씌우면 인덱스 못 씀 (인덱스는 '원본 값' 기준 정렬이라)
WHERE YEAR(created_at) = 2026      -- ✗ 인덱스 무력화
WHERE created_at >= '2026-01-01'
  AND created_at <  '2027-01-01'   -- ✓ 범위로 바꾸면 인덱스 사용

-- (2) 앞에 와일드카드가 붙은 LIKE
WHERE name LIKE '%kim'             -- ✗ (전화번호부에서 '~로 끝나는 성' 찾기 = 불가)
WHERE name LIKE 'kim%'             -- ✓ 접두어 검색은 정렬을 활용

-- (3) 타입 불일치 암시적 형변환 — phone이 문자열인데 숫자로 비교
WHERE phone = 1012345678           -- ✗ 컬럼에 CAST가 걸려 인덱스 무력화
WHERE phone = '1012345678'         -- ✓
```

> 그래서 "인덱스를 걸었는데 왜 안 빨라지죠?"의 답은 대개 **(a) 선택도가 낮거나, (b) 위 함정 중 하나로 옵티마이저가 인덱스를 못/안 쓰는 것**. `EXPLAIN`으로 실제 실행계획(`type: ref/range` = 인덱스 사용, `type: ALL` = 풀 스캔)을 확인하는 게 첫걸음이다.

## 9. 쓰기 증폭 — 인덱스를 많이 걸면 INSERT가 느려지는 이유

인덱스는 읽기를 위한 것이지만 **쓰기에는 세금**이다. 행 하나를 `INSERT` 하면:

- 테이블(클러스터형)에 1번 쓰고,
- **그 테이블의 모든 보조 인덱스 B+트리에 각각** 새 항목을 정렬 위치에 끼워 넣어야 한다.

인덱스가 5개면 `INSERT` 한 번이 **디스크 쓰기 6번**으로 불어난다. 이게 **쓰기 증폭(write amplification)**. 게다가 정렬 위치에 끼우다 리프 페이지가 꽉 차면 **페이지 분할(page split)** 이 일어나 추가 I/O와 단편화(fragmentation)까지 발생한다.

```text
INSERT 1건  →  테이블 쓰기 1 + (보조 인덱스 개수)만큼 인덱스 쓰기
            →  인덱스 5개면 실제 디스크 쓰기 ≈ 6배  (+ 페이지 분할 가능)
```

> 그래서 **"의심스러우면 일단 인덱스"는 안티패턴**이다. 쓰기가 많은 테이블(로그, 주문 이력, 큐 테이블)에 인덱스를 과하게 걸면 읽기로 번 시간보다 쓰기로 잃는 시간이 커진다. **읽기 이득 vs 쓰기·공간 비용**을 저울질해서 꼭 필요한 것만 거는 게 원칙.

## 10. 대안 구조 — 해시 인덱스와 LSM 트리

B+트리가 만능은 아니다. 워크로드가 다르면 다른 구조가 이긴다.

### 해시 인덱스 — 등치 검색 O(1), 범위는 불가

- 검색키를 **해시 함수로 버킷에 흩뿌린다**. `WHERE x = 42` 같은 **정확히 일치(equality)** 조회는 **평균 O(1)** — B+트리의 O(log n)보다 빠르다.
- 하지만 해시는 **순서를 완전히 파괴**한다. 그래서 `BETWEEN`, `>`, `ORDER BY` 같은 **범위·정렬을 아예 못 한다**. (전화번호부를 갈기갈기 찢어 무작위 서랍에 넣은 셈 — 한 명은 바로 찾지만 "가나다순 나열"은 불가)
- PostgreSQL의 `HASH` 인덱스, 그리고 §2의 `HashMap`이 이 원리. **정렬이 필요 없는 순수 등치 조회**에만 쓴다.

| | B+트리 인덱스 | 해시 인덱스 |
|---|---|---|
| 등치 조회 `= ` | O(log n) | **O(1)** (평균) |
| 범위 `BETWEEN`/`>`/`ORDER BY` | **O(log n) + 순차** | **불가** |
| 그래서 | 범용 기본값 | 등치 전용 특수 상황 |

### LSM 트리 — 쓰기 최적화 대안 (간단히)

B+트리는 데이터를 **제자리(in-place)** 에 갱신해서 정렬을 유지한다 → 쓰기마다 랜덤 I/O(§9). **LSM(Log-Structured Merge) 트리**는 발상을 뒤집는다:

- 쓰기를 우선 **메모리의 정렬 구조(memtable)에 쌓고**, 가득 차면 **디스크에 통째로 순차(append) 기록**(SSTable). 즉 랜덤 쓰기를 **순차 쓰기**로 바꾼다 → **쓰기 처리량이 매우 높다**.
- 대신 읽기는 여러 계층(memtable + 여러 SSTable)을 뒤져야 해 불리할 수 있고, 백그라운드에서 **컴팩션(compaction)** 으로 계층을 병합한다.
- 쓰기 폭주형 워크로드에 강해 **RocksDB, Cassandra, HBase, LevelDB**가 채택. Kleppmann ch.3는 이를 "B-트리(읽기·제자리 갱신) vs LSM(쓰기·추가 기록)"의 근본 대비로 정리한다.

> 정리: **읽기·범위 중심 범용 = B+트리, 순수 등치 = 해시, 쓰기 폭주 = LSM.** 워크로드가 구조를 고른다.

## 11. 자바 렌즈 — JPA/Hibernate와 인덱스, 그리고 N+1

인덱스는 SQL 밑단 이야기지만, JPA를 쓰면 **ORM이 대신 SQL을 생성**하기 때문에 개발자가 인덱스를 의식하지 못한 채 성능을 망치기 쉽다.

- **N+1 문제**: 부모 100건을 조회한 뒤(1번 쿼리), 각 부모의 자식을 지연 로딩(lazy)으로 하나씩 가져오면 **자식 조회가 100번 더** 나간다(총 1+N번). 이때 그 100번의 자식 쿼리가 **`WHERE parent_id = ?`** 인데, `parent_id`(FK)에 **인덱스가 없으면 매번 풀 스캔** → 재앙이 곱해진다. 그래서 **FK 컬럼에는 사실상 반드시 인덱스**를 건다(일부 DB는 FK에 자동 생성하지 않음). N+1의 구조적 배경과 해법(fetch join, `@BatchSize`)은 다른 노트에서 다루되, **"자식 조회 키에 인덱스가 있는가"가 N+1의 피해 규모를 좌우**한다는 점을 여기서 연결해 둔다.
- **DTO 프로젝션 → 커버링(§7)**: 엔티티 전체 대신 필요한 필드만 뽑으면 커버링 인덱스로 떨어질 여지가 생긴다.
- **`EXPLAIN`은 ORM을 못 믿을 때의 진실**: Hibernate가 실제로 어떤 SQL을 날리는지 로그로 확인하고, 그 SQL을 `EXPLAIN`에 넣어 인덱스를 타는지 봐야 한다. ORM은 편하지만 "인덱스를 무력화하는 SQL(§8 함정)"도 무심코 생성한다.

## 용어 사전

| 용어 | 뜻 |
|---|---|
| **인덱스(Index)** | 검색 속도를 위해 `(키, 행 포인터)`를 정렬해 둔 보조 자료구조. 읽기↑, 쓰기·공간↓. |
| **풀 테이블 스캔** | 인덱스 없이 테이블 전 행을 훑는 것. O(n). `EXPLAIN`의 `type: ALL`. |
| **B+트리** | 노드=디스크 페이지, high fanout으로 낮은 트리. 데이터는 리프에만, 리프는 링크드. DB 인덱스 표준. |
| **fanout(분기수)** | 노드 하나의 자식 수. 클수록 트리가 낮아져 디스크 읽기↓. |
| **페이지(Page)** | 디스크 I/O 단위(보통 4~16KB). 노드 하나가 페이지 하나에 대응. |
| **랜덤 I/O vs 순차 I/O** | 흩어진 위치를 한 번씩 읽기(느림) vs 이어서 죽 읽기(빠름). 옵티마이저 판단의 근거. |
| **클러스터형 인덱스** | 테이블 행 자체가 리프에 정렬 저장. 테이블당 1개. InnoDB에선 PK. |
| **보조(비클러스터형) 인덱스** | 리프에 PK 값만. 실제 행은 클러스터형을 되짚어 획득. |
| **되짚기(back to table)** | 보조 인덱스로 PK를 얻은 뒤 클러스터형을 다시 타 행을 가져오는 추가 조회(랜덤 I/O). |
| **복합 인덱스** | 여러 컬럼을 사전식으로 정렬한 인덱스. |
| **left-prefix 규칙** | 선두 컬럼부터 연속으로 조건이 주어져야 복합 인덱스를 탄다. |
| **커버링 인덱스** | 쿼리에 필요한 컬럼이 전부 인덱스에 있어 테이블을 안 읽는 것(index-only scan). |
| **선택도(selectivity)** | 컬럼 값의 고유함 정도. 높을수록 인덱스가 효과적. 낮으면 풀 스캔이 유리. |
| **쓰기 증폭** | INSERT 1건이 테이블+모든 인덱스 갱신으로 실제 디스크 쓰기 여러 번이 되는 현상. |
| **페이지 분할** | 리프가 꽉 찬 상태에서 삽입 시 페이지를 쪼개는 것. 추가 I/O·단편화 유발. |
| **해시 인덱스** | 등치 조회 O(1), 범위·정렬 불가. |
| **LSM 트리** | 쓰기를 순차 append로 바꾼 쓰기 최적화 구조(RocksDB/Cassandra). B-트리의 대안. |
| **EXPLAIN** | 옵티마이저의 실행계획을 보여주는 명령. 인덱스 사용 여부 진단의 출발점. |
