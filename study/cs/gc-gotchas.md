# GC 놓치기 쉬운 함정 — "된다"는 주장 vs 실제 (소스 기준)

> 관점: "요즘 GC는 좋아져서 된다"는 말이 많지만, **공식 문서·소스를 보면 조건·한계**가 붙는다. 필수 함정부터.
> Java 중심 + Unity/C#는 [../unity/deep-gc.md](../unity/deep-gc.md)(공식 매뉴얼 기반). 자바 렌즈: [../ai-ml/for-java-devs.md](../ai-ml/for-java-devs.md).

---

## 왜 이 문서가 필요한가
GC는 "자동이라 신경 안 써도 된다"고 배우지만, **성능·메모리 사고의 단골 원인**이다. "된다더라"를 **벤더 문서·JEP 원문으로 검증**하는 습관이 핵심.

---

## 핵심 함정 (필수부터)

### ① `System.gc()`는 "명령"이 아니라 "제안"
- JVM이 **무시할 수 있다**(명세상 보장 없음). 프로덕션에서 호출하면 오히려 **전체 STW**를 유발할 수 있어 **금지**. (Unity의 `GC.Collect()`도 실시간 중 금지 — 같은 교훈)

### ② GC ≠ 메모리 누수 방지
- **참조가 남아 있으면 절대 못 지운다.** static 컬렉션, 리스너 미해제, 캐시, ThreadLocal → **논리적 누수(memory leak)**. GC가 있어도 누수는 난다.

### ③ 메모리를 OS에 잘 안 돌려준다
- 힙이 한 번 커지면 **잘 안 줄어든다**(역사적 기본 동작). G1은 조건부로만 uncommit, **ZGC/Shenandoah가 반환을 개선**했지만 **기본 설정·플랫폼에 따라 제한적**.
- → **소스 기준 결론**: "피크 사용량 ≈ 상주 메모리"로 예산을 잡아라. (Unity는 더 심함 — **WebGL은 아예 반환 안 함**, → [../unity/deep-gc.md](../unity/deep-gc.md))

### ④ 저지연 GC도 "총 일"을 줄이는 게 아니다
- ZGC·Shenandoah는 **STW를 짧게 분산**할 뿐, **처리량(throughput)과 트레이드오프**가 있다(CPU·배리어 오버헤드). "무료로 빨라진다"가 아님.
- Unity **Incremental GC와 같은 원리** — 스파이크를 분산할 뿐 총량은 그대로.

### ⑤ "완전 무정지(STW 0)"는 과장
- 저지연 GC도 **짧게라도 STW 구간**이 있다(초기 마킹 등). "pause 없음"이 아니라 "pause가 매우 짧음".

### ⑥ `finalize()`는 폐기(deprecated), 쓰지 마라
- **언제 불릴지 보장 없음**(영영 안 불릴 수도). 리소스 정리는 **try-with-resources / `Cleaner`** 로.

### ⑦ OOM인데 "메모리 남아 보임"
- Java는 **압축(compaction)** 을 해서 힙 단편화는 덜하지만 — **큰 연속 배열 요구**, **Metaspace**(클래스 메타), **off-heap(다이렉트 버퍼)** 은 별개 원인. (Unity/Boehm GC는 **비압축**이라 단편화로 OOM이 더 흔함, → [../unity/deep-gc.md](../unity/deep-gc.md))

## 면접 3단 Q&A

### Q1. "요즘 GC(ZGC 등)는 멈춤이 없다"는 맞말?
- **① 결론** 과장. **멈춤이 매우 짧을 뿐**, STW 0은 아니고 **처리량을 대가로** 낸다.
- **② 원리** 동시(concurrent) 마킹·재배치로 pause를 분산. 대신 읽기/쓰기 배리어·CPU 비용 발생.
- **③ 확장** "총 작업량은 안 준다"는 게 핵심(Incremental GC와 동일). 지연 vs 처리량 선택.

### Q2. GC 언어인데 왜 메모리 누수가 나?
- **① 결론** **참조가 남아 있으면** GC가 못 지운다.
- **② 원리** 도달 가능(reachable)하면 살려둔다. static 맵·리스너·캐시가 붙잡고 있으면 도달 가능 → 영구 상주.
- **③ 확장** 해결: 약한 참조(`WeakReference`), 리스너 해제, 캐시 만료. 프로파일러로 "누가 붙잡나" 추적.

### Q3. 힙이 남는데 왜 OS 메모리는 안 줄어?
- **① 결론** GC는 보통 **힙 안에서 재사용**만 하고 **OS 반환은 소극적**.
- **② 원리** 재확장 비용을 피하려 붙잡음. 최신 GC가 개선했지만 조건부.
- **③ 확장** 예산은 **피크 기준**. 컨테이너 메모리 한도 설정 시 주의.

---

### Q4. 왜 오래 켜두면 버벅이고, 재시작하면 정상화돼? (실생활)
- **① 결론** 시간이 지나며 **살아있는 객체(live set)가 늘어** GC가 잦아지고(끊김), 메모리가 커져(스왑) 느려진다. 재시작 = **메모리 회수 + live set 리셋** → 정상.
- **② 원리 — GC가 "안 돈" 게 아니다**
  - GC는 **아무도 참조 안 하는 것(쓰레기)만** 치운다. 그런데 누수·캐시·`static`·리스너가 객체를 **계속 참조**하면, GC가 봐도 **"살아있음"으로 판단해 못 치운다** — 게을러진 게 아니라 **치울 "자격"이 없는 것**(도달 가능 = 생존).
  - 회수한 것도 힙 안에서 재사용할 뿐 **OS 반환은 소극적**(③) → 상주 메모리는 계속↑.
  - 결과: **live set↑ → GC 더 자주·길게 → 프레임 멈칫**(끊김) / **메모리↑ → 스왑 → 전체 느림**.
  ```
  오래 켬 → live set↑(누수·캐시) → GC 잦음·길어짐 → 프레임 멈칫
                              ↘ 메모리↑ → RAM 부족 → OS 스왑 → 전체 느림
  재시작 → 메모리 통째 회수 + live set 0 → 쌩쌩
  ```
- **③ 확장** 근본 원인은 **누수(참조 미해제)** — 재시작은 임시방편(서버는 주기적 rolling restart로 버팀). 발열 스로틀링·GPU 리소스 누수·단편화 등 **비(非)메모리 원인**도 섞여 재시작이 "만능약"처럼 보인다. 해결: 참조 해제·풀링·캐시 만료 (→ [../unity/deep-gc.md](../unity/deep-gc.md)).
