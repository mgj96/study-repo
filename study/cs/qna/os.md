# CS 암기 Q&A — 운영체제(OS)

> 형식: **① 결론·WHAT / ② 원리·HOW / ③ 확장·TRADE-OFF**. 외우기용. 소리 내어 3단으로 답해보기.

---

### Q1. 프로세스 vs 스레드 차이는?
- **① 결론** 프로세스는 실행 중인 프로그램(독립 메모리), 스레드는 그 안의 실행 흐름(메모리 공유).
- **② 원리** 프로세스는 코드·데이터·힙·스택을 각자 가짐. 스레드는 코드·데이터·힙을 **공유**하고 스택만 따로.
- **③ 확장** 스레드는 통신·전환이 가볍지만, 공유 자원 때문에 **동기화**가 필요(레이스 컨디션). 프로세스는 안전하나 무겁다.

### Q2. 컨텍스트 스위칭(context switching)이란? 비용은?
- **① 결론** CPU가 실행 중인 작업(프로세스/스레드)을 바꾸며 상태를 저장·복원하는 것.
- **② 원리** 현재 레지스터·PC 등을 PCB에 저장하고 다음 작업의 상태를 불러옴.
- **③ 확장** 이 자체가 **순수 오버헤드**(그 사이 일 안 함). 스레드가 코어보다 너무 많으면 스위칭만 늘어 느려짐. → 게임 서버 스레드 수 질문과 연결.

### Q3. 교착상태(deadlock)의 4가지 조건과 해결은?
- **① 결론** 서로 자원을 쥔 채 상대 걸 기다려 멈춘 상태. 4조건: 상호배제·점유대기·비선점·순환대기.
- **② 원리** 네 조건이 **동시에** 성립해야 발생.
- **③ 확장** 하나라도 깨면 예방(예: 자원 순서 정해 순환대기 제거). 또는 회피(은행원 알고리즘)·탐지 후 복구.

### Q4. 가상 메모리와 페이징이란?
- **① 결론** 물리 메모리보다 큰 주소공간을 쓰게 해주는 기법. 메모리를 **페이지** 단위로 나눠 관리.
- **② 원리** 가상 주소를 페이지 테이블로 물리 주소에 매핑. 안 쓰는 페이지는 디스크로(스왑).
- **③ 확장** 프로세스마다 독립된 주소공간·격리 제공. 단, 페이지 폴트(디스크 접근) 시 느림, TLB로 완화.

### Q5. 뮤텍스(mutex) vs 세마포어(semaphore)?
- **① 결론** 뮤텍스는 1개 자원의 잠금(소유 개념), 세마포어는 N개 자원 카운트.
- **② 원리** 뮤텍스는 잠근 스레드만 풀 수 있음. 세마포어는 카운터로 동시 접근 수를 제한.
- **③ 확장** 뮤텍스는 상호배제(임계구역)에, 세마포어는 리소스 풀·순서 제어에. 이진 세마포어는 뮤텍스와 유사하나 소유권이 없다.

### Q6. 사용자 모드 vs 커널 모드?
- **① 결론** 실행 권한 수준의 구분. 커널 모드는 하드웨어 전권, 사용자 모드는 제한적.
- **② 원리** 위험한 작업(I/O, 메모리 관리)은 **시스템 콜**로 커널 모드에 진입해 수행.
- **③ 확장** 이 분리로 앱 오류가 시스템 전체를 망가뜨리지 않음(보호). 단, 모드 전환에 비용이 있다.

### Q7. 스레드 안전(thread-safe)이란?
- **① 결론** 여러 스레드가 동시에 접근해도 결과가 올바른 상태.
- **② 원리** 공유 자원을 동기화(락·원자연산)하거나, 아예 공유하지 않게(불변·스레드 로컬) 만들어 보장.
- **③ 확장** 락이 많으면 안전하나 성능·교착 위험. 불변 객체·동시성 컬렉션으로 락을 줄이는 게 요즘 방향.

### Q8. CPU 스케줄링 알고리즘 종류는?
- **① 결론** CPU를 어떤 순서로 배분할지 정하는 규칙. FCFS·SJF·RR·우선순위 등.
- **② 원리** FCFS(먼저 온 순), SJF(짧은 작업 먼저), RR(시간 할당량 돌아가며), 우선순위(중요도 순).
- **③ 확장** 처리량·응답시간·공정성은 서로 trade-off. RR은 응답성 좋으나 스위칭 늘고, SJF는 효율적이나 기아(starvation) 위험.

### Q9. 페이지 교체 알고리즘(LRU)이란?
- **① 결론** 메모리가 꽉 찼을 때 어떤 페이지를 내보낼지 정하는 규칙. 대표는 LRU(가장 오래 안 쓴 것).
- **② 원리** LRU는 "최근 안 쓴 건 앞으로도 안 쓸 것"이라 가정해 교체.
- **③ 확장** 이상적 최적(OPT)엔 못 미치나 현실적. 완벽한 LRU는 비용이 커 근사(clock 알고리즘)로 구현.

### Q10. 동기 vs 비동기 / 블로킹 vs 논블로킹?
- **① 결론** 동기/비동기는 "결과를 기다리느냐", 블로킹/논블로킹은 "제어권을 넘기고 대기하느냐".
- **② 원리** 동기는 호출 결과를 기다려 순서대로, 비동기는 완료를 콜백·이벤트로 나중에 받음.
- **③ 확장** 비동기·논블로킹은 I/O 대기 동안 다른 일을 해 처리량↑(서버·게임). 대신 흐름이 복잡해 디버깅이 어렵다.
