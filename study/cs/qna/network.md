# CS 암기 Q&A — 네트워크

> 형식: **① 결론·WHAT / ② 원리·HOW / ③ 확장·TRADE-OFF**. 외우기용.

---

### Q1. TCP vs UDP 차이는?
- **① 결론** TCP는 신뢰성 있는 연결형, UDP는 빠른 비연결형.
- **② 원리** TCP는 연결(핸드셰이크)·순서 보장·재전송·흐름제어. UDP는 그런 보장 없이 그냥 보냄.
- **③ 확장** TCP는 웹·파일 등 정확성 우선, UDP는 실시간 게임·스트리밍·DNS 등 속도 우선. 신뢰성 vs 지연의 trade-off.

### Q2. TCP 3-way handshake란?
- **① 결론** 연결을 맺기 위해 3번 주고받는 과정(SYN → SYN-ACK → ACK).
- **② 원리** 서로의 送·수신 능력과 초기 순서번호를 확인·합의.
- **③ 확장** 이래서 TCP는 연결에 왕복 지연이 든다. 종료는 4-way handshake. (연결의 "신뢰성"은 이 절차의 대가)

### Q3. HTTP vs HTTPS 차이는?
- **① 결론** HTTPS는 HTTP에 **TLS 암호화**를 더한 것.
- **② 원리** TLS 핸드셰이크로 대칭키를 안전히 교환한 뒤, 이후 통신을 암호화. 인증서로 서버 신원 검증.
- **③ 확장** 도청·변조·위장 방지. 약간의 핸드셰이크 비용이 있으나 현대 웹의 필수(보안·SEO).

### Q4. GET vs POST 차이는?
- **① 결론** GET은 조회(데이터가 URL), POST는 생성·전송(데이터가 본문).
- **② 원리** GET은 캐시·북마크 가능하고 멱등(idempotent). POST는 본문에 데이터를 실어 서버 상태를 바꿈.
- **③ 확장** 민감 정보는 URL(GET)에 담지 말 것(로그·히스토리 노출). 규모 큰 데이터는 POST.

### Q5. DNS는 어떻게 동작하나?
- **① 결론** 도메인 이름을 IP 주소로 바꿔주는 시스템(인터넷 전화번호부).
- **② 원리** 리졸버가 루트 → TLD(.com) → 권한 서버 순으로 계층적으로 질의해 IP를 찾음.
- **③ 확장** 결과는 TTL 동안 캐싱되어 빠름. 분산·계층 구조라 확장성이 크다.

### Q6. TCP 흐름제어 vs 혼잡제어?
- **① 결론** 흐름제어는 수신자 보호, 혼잡제어는 네트워크 보호.
- **② 원리** 흐름제어는 수신 버퍼 크기(window)로 송신 속도 조절. 혼잡제어는 네트워크 상태에 따라 전송량 조절(slow start 등).
- **③ 확장** 둘 다 "너무 빨리 보내 손실나는 것"을 막지만 대상이 다름(상대 vs 경로). 재전송이 잦으면 성능이 요동친다.

### Q7. OSI 7계층 / TCP-IP 계층이란?
- **① 결론** 네트워크 통신을 역할별로 나눈 계층 모델. OSI는 7계층, TCP/IP는 4계층.
- **② 원리** 각 계층이 정해진 역할만(예: 전송·라우팅·표현) 담당하고 위아래로 데이터를 캡슐화해 전달.
- **③ 확장** 계층 분리로 한 부분을 바꿔도 나머지에 영향이 적다(모듈성). 실무는 주로 TCP/IP 모델을 씀.

### Q8. HTTP 상태 코드 분류는?
- **① 결론** 응답 결과를 나타내는 3자리 숫자. 2xx 성공·3xx 리다이렉트·4xx 클라이언트 오류·5xx 서버 오류.
- **② 원리** 첫 자리로 대분류, 예: 200 OK, 301 이동, 404 없음, 500 서버 에러.
- **③ 확장** 4xx는 요청 잘못(클라 책임), 5xx는 서버 잘못. 정확한 코드는 API 사용성·디버깅에 중요.

### Q9. 쿠키 vs 세션 vs 토큰(JWT)?
- **① 결론** 인증 상태를 유지하는 방식들. 쿠키(브라우저 저장), 세션(서버 저장), 토큰(자체 포함).
- **② 원리** 세션은 서버가 상태를 기억(쿠키에 세션ID). JWT는 서명된 토큰 자체에 정보가 담겨 서버 무상태.
- **③ 확장** 세션은 서버 부담·확장 어려움, JWT는 확장 쉬우나 발급 후 강제 만료가 어렵다. 서비스 규모로 선택.

### Q10. REST API란?
- **① 결론** 자원을 URL로 표현하고 HTTP 메서드로 다루는 API 설계 원칙.
- **② 원리** 자원(명사)을 URI로, 행위는 GET·POST·PUT·DELETE로 구분. 무상태(stateless) 통신.
- **③ 확장** 직관적·표준적이라 널리 쓰이나, 복잡한 조회엔 과다/과소 요청 문제 → GraphQL 등 대안도.
