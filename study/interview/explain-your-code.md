# 내 코드를 설명하는 법 — 면접용 프레임워크

> **핵심 메시지**: 포트폴리오는 **기능 개수**가 아니라 **설명할 수 있는 깊이**로 평가된다.
> 면접에서 **가장 점수를 많이 잃는 지점**이 바로 "이 코드 설명해보세요"에서 막히는 것.
> → 구현한 **모든 기능**에 대해 아래 3가지를 말할 수 있어야 한다.
> **관련**: [../engineering-concepts-map.md](../engineering-concepts-map.md)(설계 원칙·trade-off 렌즈)

---

## 0. 왜 중요한가

- 면접관은 "많이 만들었다"가 아니라 **"이해하고 만들었다"**를 본다.
- 기능을 100개 넣어도, 하나를 제대로 설명 못 하면 **"복붙했나?"** 의심을 산다.
- 반대로 기능이 적어도 **깊이 설명하면** "이 사람은 원리를 안다"가 된다.
- 즉, **적은 기능 + 완벽한 설명 > 많은 기능 + 얕은 설명.**

---

## 1. 3단 설명 프레임워크 (모든 코드에 적용)

### ① 무엇을 하는 코드인가 (WHAT) — 기능·목적
> "이 코드는 **무엇을** 하나요?"

- 한 문장으로 **기능·목적**을 말한다. (예: "사용자 로그인 요청을 검증하고 JWT 토큰을 발급하는 코드")
- 이게 **왜 필요한지**(어떤 문제를 푸는지)도 함께.
- ❌ 나쁜 답: "그냥 로그인이요." → ✅ 좋은 답: "인증을 담당하며, 세션 대신 토큰 방식을 써서 서버 무상태를 유지하려는 목적입니다."

### ② 어떻게 동작하는가 (HOW) — 실행 흐름·핵심 로직
> "이게 **어떻게** 동작하나요?"

- **데이터 흐름 / 실행 순서**를 단계로 설명. (입력 → 처리 → 출력)
- 핵심 로직·자료구조·알고리즘을 짚는다. (예: "요청이 필터를 거쳐 → 서비스에서 비밀번호를 해시 비교 → 성공 시 토큰 생성 → 응답 헤더에 담음")
- 왜 이 라이브러리/패턴을 썼는지도.
- ❌ "동작이 되던데요" → ✅ 단계별로 짚으며 "여기서 이 값이 이렇게 바뀝니다".

### ③ 장점과 단점 (TRADE-OFFS) — 왜 이렇게 짰나
> "이렇게 짜면 **장점·단점**이 뭔가요? 다른 방법은요?"

- **장점**: 왜 이 선택이 좋은가. (성능/유지보수/확장성/단순함)
- **단점·한계**: 무엇을 포기했나. (예: "토큰 방식은 서버가 무상태라 확장이 쉽지만, 발급 후 강제 만료가 어렵습니다")
- **대안 비교**: "세션 방식도 있는데, 이 프로젝트는 API 서버라 토큰을 택했습니다."
- 이 부분이 **합격/불합격을 가른다.** trade-off를 말하면 "생각하고 만든 사람"이 된다.

---

## 2. 재사용 체크리스트 (기능마다 한 장씩)

각 포트폴리오 기능에 대해 아래를 채워보라. 채워지지 않으면 아직 설명 준비가 안 된 것.

- [ ] **WHAT**: 이 기능을 한 문장으로 + 왜 필요한지
- [ ] **HOW**: 입력→처리→출력 흐름 3~5단계
- [ ] **HOW**: 핵심 로직/패턴/라이브러리와 그 이유
- [ ] **TRADE-OFF**: 장점 2가지
- [ ] **TRADE-OFF**: 단점·한계 1가지 + 개선 방향
- [ ] **TRADE-OFF**: 대안과 비교(왜 이걸 택했나)
- [ ] **꼬리질문 대비**: "여기서 에러 나면?", "트래픽 10배면?", "왜 A 대신 B?"

## 3. 적용 예시 (내 저장소로 연습)

내 GitHub 프로젝트에 그대로 대입해 연습하면 좋다.

| 프로젝트 | WHAT 연습 | TRADE-OFF 연습 포인트 |
|----------|-----------|------------------------|
| `market` (클린 아키텍처) | 계층 분리로 무엇을 얻나 | 클린아키텍처의 장점(테스트·유지보수) vs 단점(초기 복잡도·보일러플레이트) |
| `spring-ollama-rag` | RAG로 무엇을 하나 | 로컬 LLM(Ollama)의 장점(비용·보안) vs 단점(성능·품질) |
| `egov-boot-template` | 전자정부 설정의 목적 | 프레임워크 강제의 장점(표준) vs 단점(자유도) |
| `golang-crawling` | 크롤링 자동화 목적 | 주기적 폴링 vs 웹훅, 부하·차단 이슈 |

> **연습법**: 각 프로젝트의 **가장 핵심 기능 1개**를 골라 1장 체크리스트를 채운다. 소리 내어 3분 안에 설명해본다.

---

## 4. 흔한 실수 (감점 포인트)

- **"그냥 됐어요 / 튜토리얼 따라 했어요"** → 이해 없음으로 읽힘.
- **장점만 말하고 단점은 못 말함** → trade-off 사고 부재.
- **용어를 뜻 없이 나열** (예: "JWT 썼어요"만) → "왜?"에 무너짐.
- **전체 그림 없이 코드 줄만 설명** → 흐름(HOW)을 못 보여줌.
- **"성능 좋아요"인데 근거·측정 없음** → 측정 기반으로 말해야.

---

## 5. 한 줄 정리

> 면접은 "얼마나 만들었나"가 아니라 **"만든 걸 WHAT·HOW·TRADE-OFF로 설명할 수 있나"**를 본다.
> 기능마다 이 3가지를 준비하면, 적은 기능으로도 "원리를 아는 지원자"가 된다.
