# 동시성·락·네트워크 — 오늘치 정리 + 팁 + 퀴즈

> 오늘 공부(Redis vs DB 락, 네트워크 왕복, 인메모리 vs 디스크)를 저장 + 놓치기 쉬운 팁 +
> 직접 풀 퀴즈. **NotebookLM에 이 소스를 넣고 "이 자료로 퀴즈 내줘"** 하면 자가 테스트 가능.
> 실측: [lab-benchmarks](https://github.com/mgj96/lab-benchmarks)

## 1. 핵심 정리 (오늘치)

**5전략 (동시 재고차감):**
- `none`: 락 없음 → 빠르나 **초과판매**(정합성 실패)
- `redis-lock`: Redis 분산락 → 정합 O, **왕복 4회 + 2개 시스템**
- `db-row-lock`: `SELECT FOR UPDATE` → 정합 O, 핫로우 경합·트랜잭션 점유
- `db-atomic`: `UPDATE ... WHERE stock>0` → **락 없이 정합 + 왕복 1회 (가장 가벼움)**
- `db-split-lock`: 재고를 N행으로 쪼개 경합 분산

**왜 db-atomic이 빨랐나 (역설):**
- redis-lock도 데이터는 DB에서 처리 → Redis는 **락만**(추가 왕복). 인메모리 속도는 **디스크를 대체할 때만** 이득.
- DB는 핫 로우를 **RAM 버퍼풀**에 캐싱 → "디스크=느림"은 반만 맞음.

**네트워크 비용 = 왕복(RTT) 횟수** (전송량 아님):
- redis-lock 왕복이 큰 이유 ① DB 일에 Redis 왕복 2회 추가 ② 시스템 2개 ③ **경합 시 락 재시도로 폭증**.

**인메모리(Redis)가 이기는 임계:** 작업셋 > RAM / 수만+ ops/s·<1ms / DB가 병목 / 휘발 OK.

**db-atomic의 숨은 비용:** 부하가 전부 DB(확장 어려운 공유 자원)로 집중 → 다른 쿼리까지 느려짐.

## 2. 더 알면 좋을 팁 · 놓치기 쉬운 점

1. **락을 아예 안 쓰는 길**: `UPDATE ... WHERE stock>0`(조건부=낙관적 update)나
   **Redis 원자 명령 `DECR`**(단일 스레드라 그 자체가 원자적)로 카운터를 락 없이 처리 가능.
   → 우리 벤치엔 없던 6번째 옵션. Redis DECR은 매우 빠르나 **durability 약함**.
2. **낙관 vs 비관 락**: 저경합이면 낙관적(version/CAS)이 빠름(대기 없음). 고경합·재시도 비싸면 비관적.
3. **락 TTL 함정**: 너무 **짧으면** 작업 중 만료→두 명이 동시 보유(정합 깨짐), 너무 **길면** 죽은 락이 남을 막음.
4. **Redlock 안전성 논쟁**(Kleppmann vs antirez): 시계 스큐·프로세스 일시정지(GC 등)로 상호배제가
   깨질 수 있고 **fencing token이 없다** → **정합성 critical이면 Redlock 대신 ZooKeeper류**,
   효율 목적(중복 크론 방지 등)이면 OK.
   출처: [Kleppmann](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html),
   [antirez 반박](https://antirez.com/news/101)
5. **멱등성·유니크 제약**: 락으로 막기보다 **구조적으로 중복 불가**하게(예: 주문ID UNIQUE) 만드는 게
   종종 더 견고. "중복 결제 방지"는 락보다 멱등키가 정석.
6. **큐 직렬화**: 핫 자원을 단일 컨슈머 큐로 순서화 → 락 경합 자체를 제거.
7. **커넥션 풀 고갈**: 행락은 트랜잭션을 오래 쥠 → 고동시성에서 풀 고갈·데드락. db-atomic은 1쿼리라 안전.
8. **핫 로우는 DB 안에서도 직렬화**: 같은 행 쓰기는 내부 행락으로 줄세워짐 → 카운터 분할(shard)로 완화.

## 3. 추가 예제 (어떤 접근이 맞을까)

| 상황 | 적합한 접근 | 왜 |
|------|------------|-----|
| 쿠폰 선착순 1000장 | db-atomic 조건부 UPDATE | 재고와 동일, 단일 DB면 단순·빠름 |
| 좋아요 수(쓰기 폭주, 정합 느슨) | Redis INCR 또는 비정규화+배치 | 정확한 즉시값 불필요, DB 부하 분리 |
| 중복 결제 방지 | 멱등키 / UNIQUE 제약 | 락보다 구조적으로 견고 |
| 분산 크론 중복 실행 방지 | Redlock(효율 용도) OK | 정합성 critical 아님 |
| 샤딩된 다중 DB의 전역 락 | Redis/ZooKeeper | 단일 DB 락은 범위 밖 |

## 4. 퀴즈 (직접 풀고 → 답 펼쳐 확인 / NotebookLM에 "퀴즈 내줘")

**Q1.** redis-lock의 네트워크 왕복이 db-atomic보다 많은 이유 3가지는?

**Q2.** "Redis는 인메모리라 항상 빠르다"가 이 벤치에서 안 맞은 이유는?

**Q3.** 인메모리 저장소가 DB보다 유리해지는 임계 조건을 2가지 이상 들어라.

**Q4.** db-atomic이 빠른데도 "이게 최고"라고 단정하면 안 되는, 지표에 안 잡히는 비용은?

**Q5.** 동시성이 매우 높을 때 db-row-lock보다 db-split-lock이 나은 이유는?

**Q6.** 정합성이 절대적으로 중요한 락에 Redlock을 쓰면 안 되는 이유와 대안은?

**Q7.** 락을 아예 안 쓰고 정합성을 지키는 방법을 2가지 들어라.

**Q8.** (적용) 좋아요 수처럼 쓰기 폭주 + 정합성 느슨이면 어떤 접근이 좋고 왜?

**Q9.** 락 TTL을 너무 짧게 / 너무 길게 잡으면 각각 무슨 문제가 생기나?

**Q10.** (사고) 단일 DB·중간 트래픽 서비스라면 어떤 전략을 기본값으로? YAGNI 관점에서 설명하라.

<details>
<summary>정답·해설 (먼저 풀고 펼치기)</summary>

- **A1.** ① DB 작업 위에 Redis 락 획득/해제 왕복 2회 추가 ② Redis·DB 두 시스템 경로 ③ 경합 시 락 재시도가 Redis 왕복으로 누적.
- **A2.** redis-lock도 데이터는 DB에서 처리(Redis는 락만) → 인메모리가 디스크를 *대체*한 게 아니라 *추가*돼 왕복만 늘었다. 게다가 DB 핫로우는 버퍼풀(RAM)에 있어 사실상 메모리 접근.
- **A3.** 작업셋 > RAM(버퍼풀) / 수만+ ops/s·<1ms p99 / DB가 이미 병목(부하 분리) / 데이터 휘발 OK(세션·랭킹).
- **A4.** 부하가 전부 DB(수평 확장 어려운 공유 자원)로 집중 → 같은 DB 쓰는 다른 모든 쿼리가 느려진다.
- **A5.** 단일 핫로우는 내부 행락으로 직렬화됨. N행으로 쪼개면 경합이 ~1/N로 분산돼 동시성↑.
- **A6.** 시계 스큐·프로세스 일시정지로 상호배제가 깨질 수 있고 fencing token이 없다. 정합성 critical이면 ZooKeeper 등 합의 시스템.
- **A7.** 조건부 원자 UPDATE(`WHERE stock>0`) / 멱등키·UNIQUE 제약 / Redis 원자 DECR / 큐 직렬화 중 2개.
- **A8.** Redis INCR(원자, DB 부하 분리, 휘발 OK) 또는 비정규화+배치 반영. 정확한 즉시 정합이 불필요하고 쓰기가 많아서.
- **A9.** 짧으면: 작업 중 만료 → 두 클라이언트가 동시에 락 보유(정합 깨짐). 길면: 죽은/멈춘 보유자의 락이 남아 다른 작업을 오래 막음.
- **A10.** 모놀리스+단일 DB면 db-atomic(조건부 UPDATE)으로 시작. 지금 분산이 필요 없으니 Redis를 미리 넣지 않는다(YAGNI). 측정된 병목(DB 포화·샤딩 필요)이 생기면 그때 Redis/큐로 진화(Gall's Law).
