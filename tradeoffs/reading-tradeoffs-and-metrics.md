# 트레이드오프를 읽는 법 — 지표는 '수단'이지 절대 기준이 아니다

> 핵심 태도: 기술 선택은 "똑똑해 보이는 X/Y 그래프"가 아니다.
> **"지금 풀려는 문제가 무엇인가"** 가 먼저고, 지표·기술은 그걸 푸는 **수단**일 뿐이다.
> 무조건적인 정답은 없다.

## 1. 모든 선택에는 '축'이 있다 — 무엇을 얻고 무엇을 잃나

거의 모든 기술 결정은 **하나를 얻으면 다른 하나를 내주는** 구조다.
"좋다/나쁘다"가 아니라 **"무엇을 무엇과 바꾸나"** 로 봐야 한다.

| 선택 | 얻는 것 | 내주는 것 (대가) |
|------|---------|-----------------|
| 인덱스 추가 | 읽기(검색) 속도 | 쓰기 속도·저장공간·캐시 |
| 캐싱 | 응답 속도 | 정합성(stale)·무효화 복잡도 |
| 정규화 | 쓰기 속도·정합성 | 읽기 속도(JOIN) |
| 비정규화 | 읽기 속도 | 쓰기 속도·중복 동기화 |
| 수평 확장 | 처리량·가용성 | 지연·복잡도 |
| ORM | 생산성·이식성 | 세밀한 성능 제어 |

→ 이 표의 핵심은 값이 아니라 **사고법**: 새 기술을 볼 때 "이건 무슨 축을 바꾸지?"를 묻는다.

## 2. 왜 '인덱스'를 예로 보면 좋은가

인덱스는 트레이드오프가 **눈에 보이게** 드러나는 교과서적 예다:
- **얻음**: `WHERE`/`JOIN`/정렬 조회가 O(n)→O(log n)으로.
- **내줌**: INSERT/UPDATE/DELETE마다 인덱스도 갱신 → **쓰기가 인덱스 개수만큼 느려짐**.
  저장공간↑, 인덱스 페이지가 캐시(RAM)를 점유 → 캐시 압박↑.
- **무엇이 바뀌면 결정이 바뀌나**: **읽기:쓰기 비율**. 읽기 위주면 인덱스 이득,
  쓰기 폭주면 인덱스가 독. → "인덱스는 좋다"가 아니라 **"이 워크로드에 좋은가"**.

이 한 예시만 제대로 이해하면, 다른 모든 트레이드오프도 같은 틀로 읽힌다.

## 3. 그래서 무엇으로 검증하나 — 지표(metric)

트레이드오프를 "감"이 아니라 **숫자로 확인**하려면 지표가 필요하다. 단:

### 지표는 '문제 해결'을 확인하는 수단이지, 그 자체가 목표가 아니다
- 인덱스를 넣었으면 → **조회 p95 지연**이 줄었나? 동시에 **쓰기 처리량(TPS)**은 안 망가졌나?
- 캐시를 넣었으면 → **DB 부하/응답시간**이 줄었나? **stale로 인한 오류**는 안 늘었나?
- 즉 지표는 **"얻은 축"과 "내준 축"을 같이** 본다. 한쪽만 보면 속는다.

### 똑똑해 보이는 지표의 함정 — Goodhart's law
> "측정값이 목표가 되는 순간, 그것은 좋은 측정값이기를 멈춘다." (Goodhart's law)

지표를 **목표로 삼으면 사람들은 지표를 game**한다 (소련 못 공장: 개수 목표→쓸모없는 작은 못
양산, 무게 목표→거대한 못). 엔지니어링에서도 응답시간 평균만 쫓다 **꼬리지연(p99)**을 놓치거나,
커버리지 숫자만 채우려 의미 없는 테스트를 쓴다.
출처: [Goodhart's law (Wikipedia)](https://en.wikipedia.org/wiki/Goodhart%27s_law),
[엔지니어링 적용](https://vikky.dev/goodharts-law)

### 측정 먼저 (Knuth)
> "조기 최적화는 만악의 근원" = **측정 전에 추측으로 최적화하지 말라**.
직관은 거의 빗나간다 → 프로파일링으로 진짜 병목(3%)을 찾고 거기만.
출처: [Knuth 원문 맥락](https://probablydance.com/2025/06/19/revisiting-knuths-premature-optimization-paper/)

## 4. 한 장 요약 — 판단 순서

```
1. 무슨 문제? — "느리다"가 아니라 "무엇이, 얼마나, 누구에게 문제인가" (목적 정의)
2. 무슨 축?   — 이 선택이 무엇을 얻고 무엇을 내주나 (트레이드오프 명시)
3. 무슨 지표? — 목적을 달성했는지 + 대가가 감당 가능한지 둘 다 측정
4. 측정 먼저  — 추측 금지. 적용 전/후 비교.
5. 지표를 목표로 삼지 말 것 — 지표는 문제 해결의 증거일 뿐 (Goodhart)
```

> 결국 중요한 건 **"이 문제를 풀기 위함"**. 기술도 지표도 거기에 종속된다.
> "무조건 이게 정답"은 없다 — 이 레포의 일관된 태도.
