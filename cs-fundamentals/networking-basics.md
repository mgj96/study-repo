# 네트워크 기초 — 지연(latency) vs 대역폭(bandwidth)

> 학습용 문서. 요약 → 상세 → 실무 연결 순.

## 요약 (3줄)

- **지연(latency)** = 한 번 갔다 오는 데 걸리는 시간. **대역폭(bandwidth)** = 단위 시간당 보낼 수 있는 양.
- 둘은 다르다: 굵은 파이프(대역폭↑)라도 멀면(지연↑) 첫 응답은 느리다.
- 그래서 **요청 횟수(왕복)** 를 줄이는 게 큰 파일 한 번 받는 것보다 중요할 때가 많다.

## 상세

### 1. 지연 vs 대역폭 비유
- 지연 = 도로의 **길이**(편도 시간), 대역폭 = 도로의 **차선 수**.
- 작은 데이터를 자주 주고받으면 **지연**이, 큰 데이터를 옮기면 **대역폭**이 지배한다.
- 데이터센터 간 네트워크 왕복 ≈ 수십~수백 ms → 메모리(100ns)보다 **백만 배** 느림.
  ([메모리 계층](memory-hierarchy-and-latency.md))

### 2. 왕복(RTT)이 비싸다
- 요청 1건마다 왕복 시간이 붙는다. **요청 100개 = 왕복 100번** → 지연이 누적.
- 그래서:
  - **배칭**: 여러 요청을 하나로 묶기.
  - **N+1 쿼리 제거**: 반복 DB 왕복을 한 번에.
  - **캐싱/CDN**: 가까운 곳에 두어 왕복 거리를 줄임.
  - **HTTP keep-alive / 멀티플렉싱(HTTP/2,3)**: 연결 수립 비용 재사용.

### 3. TCP / HTTP 감각
- TCP: 연결 수립(핸드셰이크)에 왕복이 필요 → 짧은 요청 많으면 그 비용이 큼.
- HTTP는 그 위에서 동작. HTTP/2·3는 한 연결로 여러 요청을 **멀티플렉싱**해 왕복 낭비를 줄임.
- TLS(https)는 추가 핸드셰이크 → 연결 재사용이 더 중요.

### 4. 처리량의 함정
"대역폭이 충분한데 왜 느리지?" → 보통 **지연 + 왕복 횟수** 문제. 대역폭을 키워도
요청을 잘게 많이 보내면 안 빨라진다. 측정해서 어느 쪽이 병목인지 본다.

## 실무 연결 — 이걸 알면 무엇이 달라지나

- **API 설계**: 화면 하나에 요청 10번보다 **묶은 요청 1번**(또는 GraphQL/BFF)이 체감 빠름.
- **프론트 성능**: 작은 파일 수십 개보다 번들링/HTTP2로 왕복을 줄임. 이미지·정적자원은 CDN.
- **분산 시스템**: scale-out하면 노드 간 네트워크 왕복이 새 비용 → [scale-up vs out](scaling-up-vs-out.md).
- **LLM API 호출**: 토큰을 잘게 여러 번 요청하면 왕복이 누적 → 한 번에 충분한 context로
  보내는 게 유리(단 너무 크면 처리 지연·비용↑ — 트레이드오프).
