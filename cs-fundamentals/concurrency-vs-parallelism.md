# 동시성(Concurrency) vs 병렬성(Parallelism)

> 학습용 문서. 요약 → 상세 → 실무 연결 순.

## 요약 (3줄)

- **동시성**: 여러 일을 *번갈아* 다루도록 **구조화**하는 것 (한 번에 하나만 실행돼도 됨).
- **병렬성**: 여러 일을 *실제 동시에* 실행하는 것 (코어가 여러 개여야 가능).
- Rob Pike: "동시성은 **구조**, 병렬성은 **실행**. 동시성이 병렬을 *가능하게* 만든다."

## 상세

### 1. 핵심 구분 (Rob Pike, 2012)
> "Concurrency is about **dealing with** lots of things at once.
>  Parallelism is about **doing** lots of things at once."
[Concurrency is not Parallelism (go.dev)](https://go.dev/blog/waza-talk)
- 동시성 = 독립적으로 실행되는 일들의 **조합/설계**. 단일 코어에서도 성립.
- 병렬성 = 그 일들이 **물리적으로 동시에** 도는 것. 멀티코어 필요.
- 동시성 설계가 잘 돼 있으면, 코어가 늘 때 자연히 병렬로 확장된다.

### 2. 실행 모델
- **스레드**: OS가 스케줄링하는 실행 단위. 선점형(preemptive)으로 번갈아 실행.
- **이벤트 루프(단일 스레드 비동기)**: JS/Node가 대표. I/O 대기 중 다른 일을 처리 →
  **동시성은 있지만 병렬은 아님**(코어 1개). CPU 바운드 작업엔 부적합.
- **멀티프로세스/멀티스레드 + 멀티코어**: 진짜 병렬.

### 3. 함정 — 공유 상태
병렬/동시 실행이 같은 데이터를 건드리면:
- **경쟁 조건(race condition)**: 실행 순서에 따라 결과가 달라짐.
- **데드락(deadlock)**: 서로의 락을 기다리며 영원히 멈춤.
- 해법: 락(mutex), 불변 데이터, 메시지 패싱(공유 대신 통신 — Go 채널/액터 모델).

### 4. I/O 바운드 vs CPU 바운드 (선택 기준)
- **I/O 바운드**(네트워크·디스크 대기 많음): 비동기/이벤트 루프, 스레드면 충분.
- **CPU 바운드**(계산 많음): 진짜 병렬(멀티프로세스/멀티코어)이라야 빨라짐.

## 실무 연결 — 이걸 알면 무엇이 달라지나

- **Node가 빠른데 느린 이유**: 단일 스레드 이벤트 루프라 I/O는 잘 버티지만,
  무거운 계산을 넣으면 루프가 막힌다 → 워커/별도 프로세스로 분리.
- **scale-out과 연결**: 병렬은 코어/머신을 늘려야 효과 → [scale-up vs scale-out](scaling-up-vs-out.md).
- **"왜 코어 늘렸는데 안 빨라지지"**: 코드가 동시성 구조가 아니거나, 공유 락 경합이 병목.
- **AI에게 일 시킬 때**: "동시 처리해줘"는 모호 → "I/O 바운드라 비동기로" 또는
  "CPU 바운드라 멀티프로세스로"처럼 **작업 성격을 명시**해야 맞는 코드가 나온다.
