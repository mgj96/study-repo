# 프로세스 / 스레드 / OS 기초

> 학습용 문서. 요약 → 상세 → 실무 연결 순.

## 요약 (3줄)

- **프로세스** = 독립된 메모리 공간을 가진 실행 단위. **스레드** = 그 안에서 메모리를 공유하는 실행 흐름.
- OS는 한정된 CPU를 여러 실행 단위에 **번갈아 배분**(스케줄링)하고, 그때마다 **컨텍스트 스위치** 비용이 든다.
- 가상 메모리 덕분에 각 프로세스는 "메모리를 독점한 것처럼" 본다 (실제론 공유·페이징).

## 상세

### 1. 프로세스 vs 스레드
| | 프로세스 | 스레드 |
|---|---------|--------|
| 메모리 | 독립(격리) | 프로세스 내 **공유** |
| 통신 | IPC(느림·안전) | 공유 메모리(빠름·위험) |
| 장애 격리 | 강함(하나 죽어도 분리) | 약함(같이 죽을 수 있음) |
| 생성 비용 | 큼 | 작음 |

→ 격리·안정성이 중요하면 프로세스, 가볍게 많이 띄우려면 스레드.

### 2. 스케줄링과 컨텍스트 스위치
- OS 스케줄러가 어떤 실행 단위를 CPU에 올릴지 정함(선점형: 시간 할당량 소진 시 교체).
- **컨텍스트 스위치 비용**: 직접 비용 ~1-10µs(레지스터·주소공간 저장/복원).
  단 **숨은 비용**이 큼 — 스위치 후 **캐시 워밍**에 수백 µs가 들 수 있다.
  ([측정 자료](https://eli.thegreenplace.net/2018/measuring-context-switching-and-memory-overheads-for-linux-threads/))
- 스레드 스위치 < 프로세스 스위치 (주소공간/TLB를 안 바꿔도 되니까).
- → **스레드를 과하게 만들면** 스위치·캐시 무효화로 오히려 느려진다 (코어 수 근처가 적정).

### 3. 가상 메모리
- 각 프로세스는 연속된 가상 주소 공간을 보지만, 실제론 페이지 단위로 물리 메모리에 매핑.
- 물리 메모리가 부족하면 **페이징/스왑**(디스크로) → 느린 계층으로 떨어져 급격히 느려짐.
- → "메모리 부족"이 곧 "디스크 속도로 추락"을 뜻함. ([메모리 계층](memory-hierarchy-and-latency.md))

## 실무 연결 — 이걸 알면 무엇이 달라지나

- **워커 수 설정**: 웹서버 워커/스레드를 무작정 늘리면 컨텍스트 스위치로 손해.
  CPU 바운드는 ≈코어 수, I/O 바운드는 좀 더 많게 — 측정으로 결정.
- **"메모리 늘렸더니 갑자기 빨라짐"**: 스왑이 사라져 디스크 추락을 면한 것.
- **컨테이너/서버 사이징**: 프로세스 격리(안정) vs 스레드 공유(효율)의 트레이드오프를 의식.
- **동시성 코드**와 직결 → [동시성 vs 병렬성](concurrency-vs-parallelism.md).
