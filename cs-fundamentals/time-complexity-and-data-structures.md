# 시간복잡도와 자료구조 — Big-O 직관과 선택 기준

> 학습용 문서. 요약 → 상세 → 실무 연결 순.

## 요약 (3줄)

- Big-O는 **입력이 커질 때 비용이 어떻게 늘어나는지**의 증가율 (상수·작은 입력은 무시).
- 자료구조 선택 = "어떤 연산을 자주 하느냐"에 따라 결정 (검색·삽입·순회).
- 단, 빅오가 같아도 **캐시 친화성**이 실제 속도를 가른다 → 메모리 계층과 함께 봐야.

## 상세

### 1. Big-O 직관
| 표기 | 이름 | 예 | 입력 10배 늘면 |
|------|------|-----|---------------|
| O(1) | 상수 | 해시 조회, 배열 인덱싱 | 그대로 |
| O(log n) | 로그 | 이진탐색, 균형트리 | 거의 그대로 |
| O(n) | 선형 | 배열 순회 | 10배 |
| O(n log n) | 선형로그 | 좋은 정렬 | ~13배 |
| O(n²) | 제곱 | 이중 루프, 버블정렬 | 100배 |
| O(2ⁿ) | 지수 | 완전탐색 | 폭발 |

→ 핵심은 정확한 식이 아니라 "**n이 커질 때 감당 가능한가**"의 감각.

### 2. 자료구조별 트레이드오프
| 구조 | 검색 | 삽입/삭제 | 특징 |
|------|------|----------|------|
| 배열(Array) | O(n) / 인덱스 O(1) | 끝 O(1), 중간 O(n) | **연속 메모리 → 캐시 최강** |
| 해시맵(Hash) | 평균 O(1) | 평균 O(1) | 순서 없음, 충돌 시 저하 |
| 균형 트리 | O(log n) | O(log n) | **정렬 유지**, 범위 검색 |
| 링크드리스트 | O(n) | 지점 알면 O(1) | 흩어진 메모리 → 캐시 약함 |
| 스택/큐 | - | O(1) | LIFO / FIFO |

### 3. 빅오만으로 부족한 이유 (메모리 계층)
링크드리스트와 배열 순회는 둘 다 O(n)이지만, 배열이 **수배~수십배 빠르다**.
포인터를 따라가면 매번 cache miss가 나기 때문. → [메모리 계층](memory-hierarchy-and-latency.md)
"빅오가 같으면 **연속 메모리(배열) 쪽을 의심 없이 고른다**"가 실전 기본값.

### 4. 분할상환(amortized)
동적 배열 append는 가끔 O(n)(재할당)이지만 **평균은 O(1)**. 가끔 비싼 연산이
드물게 일어나면 평균으로 보는 게 맞다 (해시 리사이즈도 동일).

## 실무 연결 — 이걸 알면 무엇이 달라지나

- **"왜 느리지" 진단**: 루프 안의 루프(O(n²))나 반복되는 선형 검색(리스트에서 `includes`)이
  흔한 범인. → 자주 찾으면 Set/Map(O(1))로 바꾼다.
- **자료구조를 먼저 정하라**: 알고리즘보다 자료구조 선택이 성능을 더 크게 좌우하는 경우가 많다.
- **AI에게 요청 시**: "성능 좋게"가 아니라 "이건 10만 건을 자주 검색하니 O(1) 조회 구조로"처럼
  **연산 빈도와 규모**를 주면 정확한 선택을 한다.
